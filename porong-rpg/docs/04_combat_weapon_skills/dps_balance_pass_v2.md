## 0. 코드 진실 — 실제로 구현된 것 (grep + 파일 직독으로 확정)

### 0-1. 단일 데미지 함수는 두 개뿐이다

`BaseWeaponSkill.java`:

```java
protected double scaledDamage(ctx, player, coeff) {
    return ctx.weaponPower(player) * coeff;                       // 평탄
}
protected double scaledDamageWithStacks(ctx, player, baseCoeff, stackBonusPct) {
    int stacks = getStacks(ctx, player);
    return ctx.weaponPower(player) * (baseCoeff + stackBonusPct * stacks);   // 매끄러운 누진
}
```

**임계 배율·만충 배율은 어디에도 없다.** `dealDamage`는 일반피해%·보스피해%·크리만 곱한다. `ResourceTracker`는 `getStack`/`incrementStack(max 클램프)`/`resetStack`만 갖고, "캡 도달 시 배율" 같은 분기가 전혀 없다. 정본 §4의 "중간 임계점 효과 / 만충 효과" 컬럼은 **전부 미구현 상태의 기획 텍스트**다.

## B. 만충 스파이크 설계 (디스크리트 배율 신규 도입)

### B-1. 설계 원칙

1. **per-stack 일부를 만충 배율로 재배분**(순증 금지). 현 finisher per-stack은 이미 만충 시 보너스를 내포하므로, 만충 배율을 그냥 곱하면 이중카운팅이다.
2. **만충 시점 유효계수 = `(base' + perStack'×cap) × M`**. cap=3 기준 이 값이 밸런스 목표(아래 C)와 일치하도록 base'·perStack'을 역설계한다.
3. **비만충(부분 스택) 시 M 미적용** → `base' + perStack'×stacks`. 즉 "충전을 끝까지 채웠을 때만 폭발" 질감.
4. **대상**: 검·창·석궁·스태프 핵심기 4종만. 도끼(기절)·낫(흡혈/다음타)은 정본대로 유틸 유지 — **데미지 만충 배율 없음**.

### B-2. 재배분 산식

목표: 만충(cap3) 유효계수를 현행과 동일하게 유지하되, **per-stack을 낮춰 비만충 구간을 약화**시키고 그 손실분을 **만충 배율 M으로 되돌린다.** 이렇게 하면:
- 만충 시 총계수 불변 → C의 ±10% 수렴 목표를 깨지 않음.
- 비만충(2스택 이하) 시 약함 → "끝까지 못 채우면 약한" 스파이크 질감 부여.
- **캡6(유지형)에서 per-stack이 절반이라 폭주가 둔화**(D에서 검증).

배율 M은 **보수적으로 1.20 고정**(4무기 공통). per-stack을 절반으로 낮추고, base를 살짝 올려 cap3 총계수를 맞춘다.

만충(cap3) 목표 총계수 = C의 조정 후 핵심기 계수. 식:
```
(base' + perStack' × 3) × 1.20 = 목표총계수
perStack' = 기존 perStack / 2   (재배분: 누진 절반을 배율로 이전)
base' = 목표총계수/1.20 − perStack'×3
```

### B-3. 무기별 만충 스파이크 파라미터 (M=1.20, cap3)

C의 조정 후 목표 핵심기 계수를 먼저 정하고(아래) 역산한다. 목표 계수는 6무기 ±10% 수렴을 위한 값.

| 무기 | 현 base/perStack | cap3 현 총계수 | **목표 cap3 총계수** | M | **base'** | **perStack'** | 검산 (base'+pS'×3)×M |
|---|---|---:|---:|---:|---:|---:|---:|
| 검 결전일섬 | 3.45 / 0.12 | 3.81 | **4.20** | 1.20 | **3.32** | **0.06** | (3.32+0.18)×1.2 = 4.20 ✓ |
| 창 천뢰일창 | 3.60 / 0.08 | 3.84 | **3.70** | 1.20 | **2.96** | **0.04** | (2.96+0.12)×1.2 = 3.70 ✓ |
| 석궁 저격태세 | 4.20 / 0.10 | 4.50 | **4.20** | 1.20 | **3.35** | **0.05** | (3.35+0.15)×1.2 = 4.20 ✓ |
| 스태프 별빛쇄도 | 4.05 / 0.10 | 4.35 | **4.85** | 1.20 | **3.89** | **0.05** | (3.89+0.15)×1.2 = 4.85 ✓ |

> **per-stack을 정확히 절반**(0.12→0.06, 0.08→0.04, 0.10→0.05)으로 내려 캡6 폭주를 구조적으로 차단. base'는 cap3 목표 총계수에서 역산.
> 목표 총계수 선정 근거(C와 연동):
> - **검**: 현 8,985(−7.7%) → 핵심기 강화로 평균 근처로. 3.81→4.20.
> - **창**: 현 10,300(+5.8%) → 천뢰 약화로 억제. 3.84→3.70.
> - **석궁**: 현 11,126(+14.3%) → 쿨이 짧아 핵심기 비중이 낮으니 핵심기만으론 못 잡는다(C-2에서 별도 처리). 핵심기 총계수는 4.50→4.20으로 소폭 하향.
> - **스태프**: 현 8,088(−16.9%) → 가장 큰 보강. 4.35→4.85.

→ 석궁 11,023(+14%)·스태프 8,208(−15%)이 여전히 벗어남. **핵심기 1회/사이클만으로는 쿨 격차를 못 잡는다.** 비핵심 슬롯을 손봐야 한다.

### C-2. 비핵심 슬롯 보정 (쿨 격차 직접 대응)

석궁은 쿨이 짧아 평타·기본기 비중이 크다 → **기본기/특수기 base를 내려야** 잡힌다. 스태프는 반대로 올린다.

| 석궁 | 속사 | 0.75×3 | **0.65×3** | 사이클 5회 고빈도 → −150/사이클 |
| 석궁 | 관통볼트 | 2.20 | **2.00** | 특수기 소폭 하향 |

### C-4. ±10% 완성 2차안 (선택)

- 도끼 +0.15×80×4÷18×60 ≈ +160/분 → 9,320 (−2.8%)
- 창 −0.10×80×4÷15×60 ≈ −128/분 → 10,128 (+5.4%)

## D. 캡6(유지형) 폭주 검증

### D-1. 구조적 폭주 요인 두 가지

1. **per-stack이 3→6으로 2배.** 유지형은 `gainStack`에서 max=6.
2. **유지형은 F 발동 시 consumeStacks를 호출하지 않음** → 6스택을 유지한 채 매 핵심기 발동. 즉 **만충(6/6)이 상시**다.

### D-2. 만충 배율이 캡6에서도 켜지는가? — **이것이 핵심 리스크**

만충 판정을 `getStacks >= cap`으로 구현하면, 유지형은 cap=6이고 항상 6스택이므로 **만충 배율 M(1.20)이 매번 켜진다.** 게다가 per-stack도 6배수로 누적된다. v2 재배분 전(현행)과 비교:

| 무기 | 현행 cap6 핵심계수(배율 0) | v2 cap6 핵심계수 (perStack'×6 + M) | 배율 |
|---|---:|---:|---:|
| 검 | 3.45+0.12×6 = 4.17 | (3.32+0.06×6)×1.20 = **4.42** | +6.0% |
| 창 | 3.60+0.08×6 = 4.08 | (2.96+0.04×6)×1.20 = **3.84** | −5.9% |
| 석궁 | 4.20+0.10×6 = 4.80 | (3.35+0.05×6)×1.20 = **4.38** | −8.8% |
| 스태프 | 4.05+0.10×6 = 4.65 | (3.89+0.05×6)×1.20 = **5.03** | +8.2% |

→ **v2 재배분이 오히려 캡6 폭주를 완화한다.** per-stack을 절반으로 낮췄으므로 cap6에서의 누진분(perStack'×6)이 현행 대비 작고, M(1.20)은 곱하지만 perStack' 절감폭이 더 커서 석궁·창은 **현행보다 낮아진다.** 검·스태프만 소폭(+6~8%) 증가하나 이는 cap3 보강 목표(검·스태프를 올리기로 한 설계)의 자연 귀결이며 폭주 수준은 아니다.

### D-3. 그래도 남는 리스크와 차단책

문제는 **만충 배율이 "상시 켜진" 유지형이 cap3 소모형보다 일방적으로 유리**해질 수 있다는 점이다. 소모형은 충전→만충→소모→재충전 사이에 비만충 구간을 거치지만, 유지형은 항상 만충(M ON). 다만:

- 유지형은 **각인 고유 +%**(질풍창 +30%, 비전술사 즉발, 기계궁수 +30% 등)가 본 DPS 모델 밖에 별도로 붙는다. 만충 배율까지 상시면 이중 우위.
- **차단책 (권장):** 만충 배율 M은 **cap3(소모형 만충 기준)에서만 최대**가 되도록 하고, 유지형 cap6은 **M을 적용하지 않거나 1.10으로 약화**한다. 구현은 `M = (stacks >= 3 ? 1.20 : 1.0)`가 아니라 **`소모형이면서 stacks>=3`** 조건으로 게이팅:

```
boolean retained = engId.endsWith("_retained_01");
int cap = retained ? 6 : 3;
boolean fullCharge = getStacks >= cap;
double M = (!retained && fullCharge) ? 1.20 : 1.0;   // 유지형은 M 미적용
double coeff = (base' + perStack' * stacks) * M;
```

이렇게 하면 유지형은 **per-stack 6스택 누진만** 받고 만충 배율은 못 받는다. cap6 핵심계수:
| 무기 | 유지형 M미적용 cap6 계수 | cap3 소모형 만충 계수 | 유지형/소모형 |
|---|---:|---:|---:|
| 검 | 3.32+0.06×6 = 3.68 | 4.20 | 0.88 |
| 창 | 2.96+0.04×6 = 3.20 | 3.70 | 0.86 |
| 석궁 | 3.35+0.05×6 = 3.65 | 4.20 | 0.87 |
| 스태프 | 3.89+0.05×6 = 4.19 | 4.85 | 0.89 |

→ 유지형 핵심기 단타는 소모형 만충의 **86~89% 수준**. 유지형은 "핵심기 단타는 약하지만 자원 소모 없이 각인 고유 버프를 상시 유지"하는 트레이드오프로 정체성이 명확해진다(소모형=폭발, 유지형=지속). **이 게이팅을 권장한다.**

### D-4. v1이 지목한 검 0.15·석궁 0.12 리스크 재확인

현행 코드는 이미 검 0.12·석궁 0.10으로 하향돼 있다(DL-123 반영, reference §관찰메모). **v2는 여기서 다시 절반(0.06/0.05)으로 내리므로** cap6 per-stack 누진 폭주는 v1 우려 대비 **구조적으로 1/2.5 수준**으로 완전히 무력화된다. 검·석궁 per-stack 폭주는 v2에서 비이슈.

### E-1. BaseWeaponSkill 신규 헬퍼 (제안)

```java
/** 만충 스파이크 적용 데미지. 소모형 만충(stacks>=3)에서만 배율 M, 유지형은 누진만(D-3). */
protected double scaledDamageFullChargeSpike(SkillContext ctx, Player player,
                                             double baseCoeff, double perStack,
                                             double fullChargeMult) {
    int stacks = getStacks(ctx, player);
    String engId = ctx.playerState(player).classEngravingId();
    boolean retained = engId.endsWith("_retained_01");
    int cap = retained ? 6 : 3;
    double coeff = baseCoeff + perStack * stacks;
    if (!retained && stacks >= cap) coeff *= fullChargeMult;   // 유지형 제외
    return ctx.weaponPower(player) * coeff;
}
```

## 요약 보고

**현재 스프레드(코드 진실):** 석궁 11,126(+14.3%) ~ 스태프 8,088(−16.9%) = **37.6%**. v1의 19.3%는 미구현 임계·만충 배율로 부풀린 허구. 실제 1위 원인은 **석궁의 짧은 쿨(코드 14s, v1은 20s로 오기)**, 바닥 원인은 **스태프의 긴 쿨(20s) + 마력탄 per-stack 부재**이지 v1이 지목한 "임계 상시 ON"이 아니다(창 관통찌르기·스태프 마력탄 임계 모두 미구현 확인).

**제안 만충 배율:** 4개 데미지 핵심기(검·창·석궁·스태프)에 **공통 M=1.20** 디스크리트 스파이크. per-stack을 정확히 절반으로 내리고(0.12→0.06 / 0.08→0.04 / 0.10→0.05·0.05) base를 cap3 목표총계수에서 역산(검3.32·창2.96·석궁3.35·스태프3.89). **순증이 아닌 재배분** — 만충 cap3 총계수는 밸런스 목표값과 일치, 2→3스택에서 ~22% 디스크리트 점프. 도끼·낫은 유틸 유지(배율 없음).
