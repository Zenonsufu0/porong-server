## 개요

포로 서버는 이제 세 개의 물리 월드를 동시에 구동한다.

- **수도 메인 월드 `main_world`** — 기존 메인 월드 콘텐츠. 수도·외곽 필드보스·튜토리얼 존·수도 반경 WorldGuard 제한 유지.
- **영지 월드 `estate_world`** — 플레이어별 개인 영지 전용 월드. Lv1~5 크기(2×2 ~ 6×6 청크) 각 플레이어 독립 구획. 설비·생산·청사진 블록 공간.
- **체험 월드 `experience_world`** — 기존 체험 월드(최초 접속 격리 공간) + 2026-04-19 결정으로 **3강등 상태(디스코드 미연동 / 리소스팩 미적용 / v1.1 재동의 대기) 공용 대기실**로 진화.

본 에픽은 세 월드 사이의 **이관 UX·WorldGuard 재설계·영지 월드 맵 생성·체험 월드와의 서사 연결·데이터 일관성**을 기획자 관점에서 1차 정리한다. 기존 기준(영지 영속 보존·영지 주인 단독 블록 조작·청크 단위 관리·WorldGuard 격리·체험 월드 경량화)을 **최대 재활용**하며, 과한 재설계는 금지한다.

---

## 목표

1. **세 월드 간 이관 동선**을 유저 혼란 없이 정리한다(수도 ↔ 영지, 수도 ↔ 체험, 영지 ↔ 체험 세 경로 전부).
2. **WorldGuard 리전·per-player WorldBorder 기존 설계를 최대한 보존**하면서 3월드 각각 독립 리전 세트로 확장한다.
3. **영지 월드 맵 생성 방식**의 선택지 3안을 기획자 관점에서 비교하고 권장안을 제시한다(엔지니어링 세부는 별도 위임).
4. **영지 청크 배치 알고리즘**의 기획 요구사항(플레이어당 독립 영역·Lv5 확장 여유·시즌 영속성 동시 만족)을 정의한다.
5. **체험 월드 ↔ 영지/수도 이관 UX**를 3강등 공용 대기실 정책과 일관되게 정리한다.
6. **이관 데이터 일관성**(위치 저장·복귀 좌표·인벤토리·크래시 복구·영지 동시 편집 잠금)의 기획 요구사항을 제시한다.

## 핵심 규칙

- **영지 주인 단독 블록 조작**은 영지 월드 전역에 유지. 방문자는 구경·통로 이동만.
- **영지는 시즌 리셋에도 영속 보존**(물리 구조물·월렛·생산 큐 전부). 영지 월드는 계정(UUID) 귀속 청크 구획의 집합.
- **수도 반경 제한**은 수도 메인 월드 전용 규칙. 영지 월드·체험 월드에는 적용하지 않는다(무의미).
- **체험 월드는 계정 데이터 미생성 원칙**을 유지하되, 3강등 공용 대기실 역할을 위해 "복귀 대상 월드(수도 or 영지)" 상태 플래그만 세션에 보관한다.
- **이관은 항상 `player.teleport()` 단일 API**를 통해 실행한다. Bungee·서버 이동 없음.
- **영지 월드 맵은 바닐라 월드 생성기를 호출하지 않는다**(체험 월드 원칙과 동일). 플랫·보이드·스키매틱 기반 중 선택.
- **동일 UUID의 크로스 기기 동시 접속 방지 로직이 선행**되어야 영지 청크 "단일 로드" 가정이 성립한다(축 1 리스크).

## 구현 포인트

### 1. 월드 간 이관 UX

#### 1-A. 수도 ↔ 영지 (신규 경로)

기획자 관점 제안:

- **진입 트리거 3종 병행**: (1) 명령어 `/영지`(기본), (2) 수도 내 "영지 관리 NPC"(상업 지구 배치, 명령어 미숙 유저 대응), (3) 수도 특정 포털 블록("영지 청사진 포털", 영지 Lv1 부여 완료 후 활성).
- **복귀 좌표 저장**: 영지 진입 직전의 수도 월드 좌표를 "영지 복귀 좌표"로 저장. 영지 내부 `/복귀` 또는 영지 출구 NPC 상호작용 시 해당 좌표로 `player.teleport()`.
- **로딩 연출**: 체험 월드 이관과 동일 기조(페이드 2초). 영지 첫 진입 시만 "영지의 문이 열립니다" 서사 1줄 + 이후 방문은 연출 최소화.
- **이관 대상 월드 결정**: 수도에서 `/영지` 호출 시 플레이어 UUID 귀속 영지 좌표를 조회 → 영지 월드 내 해당 좌표로 `player.teleport()`.

| 트리거 선택지 | 장점 | 단점 |
|---|---|---|
| A. 명령어 단일(`/영지`) | 구현 간단, 남용 방지 | 신규 유저 학습 비용, 몰입 저해 |
| B. 명령어 + NPC 2중 | UX 자연스러움, 명령어 백업 | NPC 배치·리소스 1명 추가 |
| **C. 명령어 + NPC + 포털 3중 (권장)** | 몰입·접근성·서사 모두 확보, 포털은 영지 Lv1 부여 퀘스트 서사와 연동 | 구현 범위 소폭 증가, 포털 블록 스키매틱 필요 |

권장: **C안**. 영지 Lv1 부여 퀘스트 클리어 시 "영지 청사진 포털"이 수도 상업 지구에 활성화되는 서사 연결까지 포함.

#### 1-B. 수도 ↔ 체험 (기존 결정 유지)

- 최초 접속: 자동 체험 월드 스폰(변경 없음).
- `/confirm <OTP>` 완료: 수도 메인 월드 스폰으로 `player.teleport()`(변경 없음).
- v1.1 재동의 완료: 체험 월드에서 수도 복귀(변경 없음).
- 본 에픽에서는 기존 UX를 **재확인**만 하고 수정하지 않는다.

#### 1-C. 영지 ↔ 체험 (신규 경로)

기획자 관점 제안:

- **리소스팩 미적용 상태로 영지 접속 시도 시 자동 체험 이동**: 영지 월드는 커스텀 청사진 블록·2x2 비주얼이 리소스팩 의존 가능성 높음 → 리소스팩 미적용자가 영지 진입 시 비주얼 깨짐 위험. 진입 전 검사 → 미적용이면 체험 월드로 라우팅 + 문지기 NPC가 "리소스팩 재다운로드" 안내.
- **체험 → 영지 복귀**: 리소스팩 적용 확인 시 문지기 NPC가 "영지로 돌아가시겠습니까?" 선택지 제공. 복귀 좌표는 체험 강등 직전 영지 좌표.
- **디스코드 미연동·v1.1 재동의 대기 강등 케이스**: 영지에서 강등 발생 시에도 동일 — 체험 월드로 `player.teleport()` + 복귀 대상 플래그에 "영지" 저장.

#### 1-D. 방문자가 타 플레이어 영지 방문

기획자 관점 제안:

- **주인 초대 기반**: 주인이 온라인 상태에서 `/초대 <플레이어>` 명령어 발급 → 상대에게 수락 UI 팝업 → 수락 시 `player.teleport(주인 영지 스폰)`.
- **화이트리스트**: 주인이 `/영지허용 <플레이어>` 등록 시 상시 방문 허용(주인 오프라인이어도). Lv3 이상 영지에서만 해금(접근성 게이트).
- **방문 쿨다운**: 동일 영지 재방문 5분 쿨다운(스팸 방지). 주인은 본인 영지 쿨다운 없음.
- **주인 오프라인 시 기본 정책**: 화이트리스트 미등록 상대는 방문 불가. 영지 청크 언로드 정책(청크 부하 자문 축 1)과 일관.

| 방문 정책 선택지 | 장점 | 단점 |
|---|---|---|
| A. 초대 단일(주인 온라인 필수) | 구현 간단, 남용 없음 | 오프라인 친구 영지 구경 불가 |
| **B. 초대 + 화이트리스트 2중 (권장)** | 친구·길드 구경 수요 흡수, 청크 부하 관리 | 화이트리스트 UI 추가 |
| C. 공개 영지 모드 | 쇼케이스·커뮤니티 활성화 | 청크 부하 폭발, 범위 외 |

권장: **B안**. 화이트리스트는 Lv3 이상 해금 게이트.

### 2. 수도 반경 제한 정합성

기존 수도 반경 WorldGuard 제한(`poro_factory_estate_draft.md` 수도 반경 동선 섹션)은 **수도 메인 월드 전용**으로 재정의한다.

- **영지 월드**: "수도 반경"이라는 개념 자체가 무의미. 영지 월드는 플레이어별 구획의 집합이므로 수도 관할 경계가 존재하지 않음.
- **영지 월드에서 수도 외곽 탈출 시도 처리**: 영지 월드에는 수도 외곽이 없으므로 이 시도 자체가 발생하지 않음. 영지 월드의 경계는 "자신의 영지 청크 경계"로, per-player WorldBorder로 처리(항목 3과 연계).
- **수도 반경 해제 전 영지 월드 진입 허용 여부**: 영지 월드 진입은 수도 반경 해제와 무관 — **영지 Lv1 부여 퀘스트 클리어만 게이트**. 수도 반경 해제(메인 7 완료)는 수도 메인 월드 외곽 지역 접근용이며 영지 월드와 서로 독립.

| 수도 반경 정합성 선택지 | 장점 | 단점 |
|---|---|---|
| A. 영지 월드에도 수도 반경 개념 적용 | 일관성 | 무의미·혼란 |
| **B. 수도 메인 월드 전용으로 재정의 (권장)** | 단순·직관 | 없음(기존 설계와 충돌 없음) |

권장: **B안**. 영지 월드의 경계는 per-player WorldBorder로 별도 처리.

### 3. WorldGuard 리전 재설계

3월드 각각 독립 리전 세트로 구성한다.

- **수도 메인 월드**: 기존 리전 유지(수도 외곽·외곽 필드보스·중앙 광장 등) + 수도 반경 per-player WorldBorder + 외곽 탈출 차단 flag.
- **영지 월드**:
  - 월드 전역 기본 리전 `estate_global`: PVP 차단, 몹 스폰 차단, 날씨 고정, 외부 경제 이벤트 차단.
  - **개별 영지 자동 생성 리전 `estate_<uuid>`**: 영지 할당 시 플러그인이 자동 생성. 주인만 `block-break / block-place / interact` 허용. 방문자는 `pass-through + interact: deny`.
  - per-player WorldBorder: 영지 주인은 자기 영지 경계(Lv별 2×2 ~ 6×6 청크)로 설정, 방문자는 방문 중인 영지 경계로 설정.
- **체험 월드**: 기존 단일 리전 `exp_all` 유지(전투·경매·영지·랭킹 차단). 3강등 공용 대기실 정책 연동 — 리전 플래그 변경 없이 문지기 NPC 대사/라우팅 분기.

| WorldGuard 재설계 선택지 | 장점 | 단점 |
|---|---|---|
| A. 3월드 모두 동일 리전 템플릿 | 운영 단순 | 월드별 규칙 차이 미반영 |
| **B. 월드별 독립 세트, 영지 월드만 자동 생성 리전 (권장)** | 기존 수도/체험 리전 최대 재활용, 영지만 신규 확장 | 영지 리전 자동화 플러그인 훅 필요 |
| C. 리전 대신 플러그인 훅 단독 | 리전 오버헤드 제거 | WorldGuard 에코시스템·관리자 도구 상실 |

권장: **B안**. 기존 수도/체험 리전은 변경 없음, 영지 월드만 자동 생성 리전 확장.

### 4. 영지 월드 맵 생성 방식 (최대 리스크)

본 항목이 본 에픽의 가장 큰 엔지니어링 리스크 — **플레이어당 독립 영역·Lv5 확장 여유·시즌 간 영속성**을 동시에 만족해야 한다.

기획자 관점 요구사항:

- 영지 외부 공간은 **존재하지 않거나 비활성** 상태여야 함(바이옴 탐험·야생 몹 조우 같은 "영지 외 활동"은 금지).
- 영지별 기본 지형은 **플랫·고른 바닥**이 기본이지만, 바이옴 프리셋(목재 채집기 등) 연동 위해 영지마다 "바이옴 태그"가 있어야 함.
- Lv5 확장(6×6 청크) 여유 공간이 사전 확보되어야 함(Lv1 유저 옆에 Lv5 유저가 이미 앉아있으면 확장 불가).
- 시즌 리셋에도 영지 위치 고정(계정 UUID 귀속 좌표 불변).

선택지 3안 비교:

| 맵 생성 방식 | 공간감 | Lv5 확장 여유 | 시즌 영속성 | 개발 난이도 | 리스크 |
|---|---|---|---|---|---|
| A. 플랫 월드 + 격자 배치(스폰 순서대로 부채꼴 확장) | 중간. 플랫 바닥 위 영지 구획 | 격자 칸당 Lv5 사전 예약(7×7 청크 슬롯에 6×6 영지) | 우수 (좌표 고정) | 중 (격자 할당 알고리즘) | 플랫 지형 획일성, 바이옴 표현 제한 |
| **B. 보이드 월드 + 영지 스키매틱 동적 생성 (권장)** | 우수. 영지 경계 밖은 공허로 몰입 집중 | 격자 슬롯 Lv5 기준 예약 + 보이드 공간은 확장 비용 없음 | 우수 (좌표 고정) | 중상 (보이드 생성기 + 스키매틱 로더) | 보이드 낙하 방지 월드보더 필수 |
| C. 커스텀 지형 월드 + 자연스러운 배치 | 우수. 바이옴·지형 다양 | 지형 간섭으로 Lv5 확장 시 인접 지형 침범 가능 | 중간 (지형 변형 누적) | 고 (지형 생성기 + 간섭 해결) | 확장 충돌 빈번, 영속 보존 복잡 |

권장: **B안 (보이드 월드 + 영지 스키매틱 동적 생성)**.

근거:
- 영지 외부 공간은 애초에 "존재하지 않는" 것이 설계 철학과 가장 일치(영지 = 전용 공간). 보이드는 이 철학을 시각적으로도 강화.
- 격자 슬롯 예약이 단순 — 7×7 청크 슬롯 하나에 6×6 영지 최대 확장 + 1청크 여유. 슬롯 경계에서 영지 간 충돌 원천 차단.
- 바이옴 다양성은 "스키매틱 바이옴 프리셋"으로 해결 가능(설치 시 스키매틱 선택).
- 보이드 낙하 방지는 per-player WorldBorder + 영지 경계 벽 블록으로 이중 차단.

엔지니어링 세부(스키매틱 로더·보이드 생성기·슬롯 할당 로직)는 **implementation-reviewer 2단계 위임**.

### 5. 영지 청크 배치 알고리즘

기획자 관점 요구사항(엔지니어링 세부 제외):

- **신규 플레이어 영지 생성 시 월드 좌표 할당**: 스폰 순서대로 부채꼴/나선 방식으로 다음 빈 슬롯 할당. 랜덤 좌표 예약은 파편화 위험.
- **Lv5 확장 여유 사전 예약**: 영지 Lv1 부여 시점에 이미 7×7 청크 슬롯을 할당받음. Lv 증가 시 새 좌표 할당 없이 같은 슬롯 내에서 경계 확장.
- **플레이어당 월드 좌표 고정**: 계정 UUID 귀속. 리스폰·재접속·시즌 리셋 모두 좌표 불변.
- **영지 해제 발생 시**: 계정 삭제·장기 미접속 등 극단 상황에서만 슬롯 회수 고려. 회수 시에도 **즉시 재사용 금지, 시즌 경계에서 일괄 재활성화**(데이터 오염 방지). 기본 정책은 "봉인 유지".

| 청크 배치 선택지 | 장점 | 단점 |
|---|---|---|
| A. 스폰 순서 부채꼴 확장 | 구현 간단, 파편화 없음 | 초기 유저 중심, 후기 유저는 먼 좌표 |
| **B. 부채꼴 + Lv5 슬롯 사전 예약 (권장)** | A 장점 + 확장 충돌 원천 차단 | 슬롯 고정 크기로 저레벨 영지도 7×7 슬롯 점유(공간 비효율 허용 가능) |
| C. 무작위 좌표 예약 | 유저별 고유감 | 파편화, 확장 충돌, 할당 탐색 비용 |

권장: **B안**. 엔지니어링 세부(슬롯 테이블 스키마·할당 트랜잭션·회수 SLA)는 implementation-reviewer 위임.

### 6. 체험 월드와의 인접성·서사 연결

기획자 관점 제안:

- **체험 월드 강등 UX 확장**: 기존 최초 접속 외에 3강등 경로(디스코드 미연동·리소스팩 미적용·v1.1 재동의 대기) 모두 체험 월드로 `player.teleport()`. 문지기 NPC가 상태 플래그 감지해 대사/복귀 플로우 분기. `poro_experience_world_level_design_draft.md` 오픈 질문("v1.1 재동의 강등 복귀 플로우")의 후속 작업을 본 에픽과 병행.
- **복귀 대상 월드 플래그**: 체험 강등 시 "복귀 대상" 필드에 `main_world` 또는 `estate_world` 저장. 영지 월드에서 강등된 경우 복귀 대상 = `estate_world` + 복귀 좌표.
- **영지 월드 존재를 체험 월드에서 언급할지**: **언급 최소화** 권장. 체험 월드의 세계관 톤(1막 메인 프롤로그 전단계)과 영지(생활 시스템)는 서사적 레이어가 다름. 언급하면 "영지 = 기획적 안전장치" 인상이 강해져 몰입 저해. 문지기 NPC가 "너의 자리" 정도의 추상 표현으로만 암시.

| 서사 연결 선택지 | 장점 | 단점 |
|---|---|---|
| A. 영지 월드 완전 미언급 | 몰입 보호 | 3강등 복귀 플로우에서 유저 혼란 가능 |
| **B. 추상 암시 + 문지기 NPC 분기 대사 (권장)** | 몰입 보호 + 복귀 안내 명확 | NPC 대사 분기 설계 필요 |
| C. 영지 언급 명시 | 복귀 안내 완벽 | 서사 레이어 혼선 |

권장: **B안**.

### 7. 이관 중 데이터 일관성

기획자 관점 요구사항(엔지니어링 세부 제외):

- **플레이어 위치 저장·복귀 좌표**: 이관 직전 "출발 월드·좌표·시각" 기록. 복귀 시 해당 좌표로 `player.teleport()`.
- **인벤토리 동기화**: 월드 이관 시 인벤토리는 계정 단위 공유 유지(체험 월드 제외 — 체험 월드는 인벤토리 비우기). 수도 ↔ 영지는 인벤토리 연속성 보장.
- **이관 도중 서버 크래시 복구**: 이관 상태 플래그(`teleporting=true`)를 DB에 기록, 복구 시점에서 해당 플래그 탐지 → 복귀 좌표로 복원 + 경고 로그. 영지 Lv 업그레이드 실패 복구 정책(재화 자동 재지급 → 디스코드 문의 안내 → 운영 수동)과 동일 3단 원칙.
- **영지 월드 점유 잠금**: 동일 영지 동시 편집 방지. 주인 접속 중 영지 리전은 "주인 단독 편집 락" 상태 — 방문자는 구경만. 크로스 기기 동시 접속 방지 로직 선행 필요(축 1 리스크).

| 데이터 일관성 선택지 | 장점 | 단점 |
|---|---|---|
| A. 이관 상태 플래그 없음, 최종 저장만 | 구현 간단 | 크래시 시 위치 소실 |
| **B. 이관 상태 플래그 + 3단 복구 (권장)** | 영지 업그레이드 실패 복구와 원칙 일관 | DB 쓰기 1~2회 추가 |
| C. 분산 트랜잭션 | 완벽 | 과한 엔지니어링 |

권장: **B안**. 엔지니어링 세부(트랜잭션 스키마·크래시 탐지·복구 SLA)는 implementation-reviewer + data-schema 위임.

## 오픈 질문

1. **영지 월드 맵 생성 방식 권장안(B 보이드 + 스키매틱 동적 생성) 확정 여부**: A·C 대비 본 문서 권장 근거가 타당한지. 보이드 월드에 대한 심미적 반응이 우려되는 경우 A(플랫)로 후퇴할 여지.
2. **Lv5 슬롯 사전 예약(7×7 청크) 크기 적정성**: 여유 1청크로 충분한지, 2청크 이상 확보할지. 인접 영지 간 시각적 거리감과 맵 파일 성장 속도 트레이드오프.
3. **크로스 기기 동시 접속 방지 로직 선행 여부**: 본 에픽은 이를 전제로 하지만 구현 우선순위가 미정. 이관 락 설계가 이 로직 없이 성립할 수 있는지 검토 필요.
4. **수도 내 "영지 청사진 포털" 배치 위치**: 상업 지구 중앙 광장 근처가 자연스러우나 기존 수도 맵과 충돌 없는지 확인. 수도 맵 기획자(있다면) 협의.
5. **방문자 화이트리스트 해금 레벨**: 권장 Lv3이 적정한지, Lv2 또는 Lv1부터 해금할지. 접근성 vs 청크 부하 밸런스.
6. **영지 월드 바이옴 프리셋 개수**: 스키매틱 기반 바이옴 다양성을 몇 종 지원할지(기본 3~5종 권장, 목재 채집기 연동 기준).
7. **체험 월드 문지기 NPC 대사 분기 설계 범위**: 3강등 상태별 대사 분기를 본 에픽에서 확정할지, `poro_experience_world_level_design_draft.md` 후속 2차에서 확정할지.
8. **영지 해제(계정 삭제 등) 시 슬롯 회수 정책**: 시즌 경계 일괄 재활성화 vs 영구 봉인. 장기 운영 데이터 누적 영향.
9. **영지 월드 진입 시 로딩 연출 길이**: 첫 진입 페이드 2초 + 서사 1줄 + 이후 최소화 기조가 적정한지.
