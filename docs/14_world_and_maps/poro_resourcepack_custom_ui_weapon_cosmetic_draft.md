> 작성일 기준: 2026-04-19

## 기존 포로 서버 기준과의 충돌 점검

먼저 CLAUDE.md 원칙에 따라, 본 설계가 기존 확정 기준과 충돌할 수 있는 지점을 모두 나열한다. 확정되지 않은 항목은 "오픈 질문"으로 이월한다.

- **장비 축(T1/T2, 5세트, 잠재 3줄, 공용각인 4슬롯, 강화 25강)**: 본 설계는 이 축을 그대로 유지한다. 리소스팩은 외형 레이어만 덧붙이며, 기능/수치/슬롯 수를 변경하지 않는다.
- **올인원 조합대 단일 제작 UI**: 기존 확정(`03_life_system_core/poro_allinone_workbench_draft.md`)을 유지. 커스텀 UI는 해당 UI를 렌더 수단으로 확장한다.
- **바닐라 조합 UI 전면 차단 + 치장 블록은 수도 상점 NPC 판매로 격리**: 기존 확정 유지. 본 설계는 이 격리를 어기지 않는다.
- **Citizens 5 NPC 체험 월드 C안 (click-handler 수준)**: 기존 확정 유지. 리소스팩은 NPC 외형 스킨에만 관여하며, 대화 엔진 복원을 요구하지 않는다.
- **CustomModelData 기반 커스텀 블록(2×2 청사진 블록 검토 중)**: 기존 방향 유지. 본 설계는 같은 CustomModelData 번호 공간을 사용하므로 **번호 충돌 방지용 전역 레지스트리**가 반드시 필요하다 — 이것이 본 문서의 핵심 선제 과제다.
- **EmpireRPG 커스텀 플러그인 단독 판정**: 기존 확정 유지. 리소스팩은 클라이언트 자원일 뿐이며 판정 주체는 변하지 않는다.

충돌 지점 없음. 다만 **CustomModelData 번호 충돌**과 **리소스팩 배포 방식(서버 강제 vs 수동)** 두 가지는 오픈 질문으로 정리한다.

## 1. 리소스팩 전체 구조 (상위)

### 1.1 개요
포로 서버 전용 클라이언트 리소스팩을 구축해 커스텀 UI, 무기 외형, 치장 아이템을 일관된 시각 축에서 제공한다.

### 1.2 목표
- 클라이언트 외형만으로도 "포로 서버다"가 식별되도록 한다.
- CustomModelData 번호를 체계화해 충돌과 마이그레이션 리스크를 줄인다.
- 시즌 업데이트 시 자원 추가/교체를 버전 관리로 안전하게 한다.

### 1.3 핵심 규칙
- **패키지 네이밍**: `poro-resourcepack-s{시즌}-v{major}.{minor}.{patch}` (예: `poro-resourcepack-s1-v0.1.0`).
- **semver 원칙**: major = 자원 구조/번호 체계 변경, minor = 신규 자원 추가, patch = 텍스처/모델 수정.
- **CustomModelData 번호 체계 (초안)**: 7자리 숫자로 카테고리·직업·등급·인덱스를 인코딩.
  - `ABCDDEE` 형태 제안
    - `A` = 대분류 (1=장비무기 / 2=치장 / 3=UI 글리프 / 4=블록 / 5=소비품 / 9=예비)
    - `B` = 직업/계열 (1=전사, 이후 추가 확장)
    - `C` = 무기종/카테고리 (1=대검 / 2=태도 / 3=창 / 9=공용·비무기)
    - `DD` = 등급·티어 (10=T1 기본, 20=T2 기본, 30~=치장 변형, 99=이벤트)
    - `EE` = 연번 (00~99)
  - 예: `1112010` = 장비무기/전사/대검/T1/연번00
  - 예: `2019901` = 치장/공용/비무기/이벤트/연번01
  - **번호 예약 구간**:
    - 1000000~1999999: 장비·무기
    - 2000000~2999999: 치장
    - 3000000~3099999: UI 폰트 글리프
    - 4000000~4099999: 커스텀 블록
    - 5000000~5099999: 소비품(음식·물약 등)
    - 9000000~9999999: 예비/임시
- **전역 레지스트리**: CSV 또는 YAML로 `customModelData → 용도 → 자원경로 → 소유 도메인` 1:1 매핑을 관리. 커밋 단위로 PR 리뷰.
- **배포 방식 = 3단계 확정 (2026-04-19 사용자 결정)**:
    1. **1단계 자동 다운로드**: 서버 입장 시 Paper 자동 리소스팩 프롬프트(`require-resource-pack: true` 기반). 최신 URL·해시는 서버 부팅 시점에 결정.
    2. **2단계 수동 복구**: 1단계 실패 시 봇이 해당 유저에게 **디스코드 DM**으로 최신 리소스팩 URL·해시·수동 설치 가이드 발송(Paper `resource-pack.yml` 실패 이벤트 훅 + 디스코드 봇 연동). 수동 적용 후 재접속 시 2단계 pass.
    3. **3단계 미적용자 체험 월드 자동 이동**: 1·2단계 모두 미해결(리소스팩 미적용) 상태로 접속한 플레이어는 **체험 월드로 자동 텔레포트**. 메인·영지 월드 접근 차단. 적용 확인 시 즉시 복귀. 체험 월드가 "디스코드 미연동 / 리소스팩 미적용 / v1.1 재동의 대기" 3개 강등 상태 공용 대기실.
    **후속 결정 반영 (2026-04-19 사용자 승인)**:
    - **용량 상한**: "서버 무리 안 가는 초기치 + 추후 확장성" 원칙. 구체 수치는 implementation-reviewer 후속 검토(초기 권고 100MB 권역, 확장 여지 확보). 7자리 CustomModelData 인코딩으로 번호 확장성은 충분.
    - **버전 업데이트 감지 로직 = 도입 확정**: 서버 부팅 시 `resource-pack.yml` 해시 갱신 → 클라이언트 접속 시 해시 불일치 감지 → 자동 다운로드 재프롬프트(1단계 재실행). 해시 불일치 지속 시 2단계 디스코드 수동 경로 → 3단계 체험 월드 강등으로 순환.

### 1.4 1차 범위
- 패키지 네이밍 규칙 확정.
- CustomModelData 7자리 인코딩 규칙 + 예약 구간표 확정.
- `assets/poro/` 이하 폴더 구조 표준화 (models / textures / font / sounds / lang).
- 전역 레지스트리 CSV 1파일 생성 및 커밋 워크플로 수립.

### 1.5 2차 이후
- 여러 리소스팩 호스팅(메인 + 옵션 팩) 및 옵션별 on/off.
- 다국어 lang 파일 분리(한/영).
- 시즌 아카이빙 정책(이전 시즌 자원 보존 경로).

### 1.6 비범위 (1차)
- 실제 텍스처/모델 제작 명세.
- CDN/다운로드 서버 구축.
- 무결성 해시 갱신 자동화 파이프라인.

### 1.7 폴더 구조 (초안)
```
assets/poro/
  models/
    item/          # 무기·치장·소비품 아이템 모델
    block/         # 커스텀 블록 모델
  textures/
    item/
    block/
    gui/           # GUI 배경 텍스처
    font/          # 폰트 글리프 원본
  font/
    default.json   # 폰트 글리프 매핑 (U+E000 이후 PUA 영역)
  sounds/
    ui/
    weapon/
  lang/
    ko_kr.json
```

## 2. 커스텀 UI (메뉴·GUI)

### 2.1 개요
플러그인이 여는 모든 주요 GUI 화면을 리소스팩 글리프 기반으로 꾸민다. 바닐라 더블체스트 타이틀에 투명 폰트 글리프로 배경 이미지를 겹치는 "Unicode font trick" 방식을 1차로 채택한다.

### 2.2 목표
- 텍스트 타이틀이 아니라 그래픽 배경으로 각 UI 정체성을 준다.
- 아이콘·버튼은 폰트 글리프 1글자로 치환 가능한 구조로 만든다.
- UI 간 공용 글리프(뒤로가기·확인·취소 등)를 최대한 재사용한다.

### 2.3 핵심 규칙
- **렌더 방식**: 챗 컴포넌트에 Unicode PUA(U+E000~U+F8FF) 글리프 + 음수 스페이스를 조합해 배경 이미지를 타이틀 영역에 그린다.
- **글리프 네이밍**: `poro:ui/<ui_code>/<element>` (예: `poro:ui/wallet/bg_main`).
- **공용 글리프 세트**: 뒤로가기, 확인, 취소, 페이지 이동, 잠금, 새로고침, 알림 배지 7종 우선 확정.
- **UI 코드 체계**: 2자리 도메인 + 2자리 화면 (예: `WB01` = Workbench 메인, `WA01` = Wallet 메인).

### 2.4 1차 UI 목록 (7종)
1. 올인원 조합대 메인 (WB01)
2. 월렛/지갑 (WA01)
3. 인벤토리 보조 GUI (장비 슬롯 프리뷰) (IV01)
4. 영지 관리 허브 (ES01)
5. 경매장 메인 (AU01)
6. 퀘스트 저널 (QJ01)
7. Shift+좌클릭 Shortcut Menu (SM01)

### 2.5 2차 이후 UI (3~4종)
- 차원도서관(아이템 도감)
- 약관/공지 뷰어
- 칭호·치장 장착 관리
- 상점 NPC 공용 거래 UI (1차는 바닐라 머천트 유지 가능성 있음)

### 2.6 1차 범위
- 공용 글리프 7종 확정.
- 1차 UI 7종 각각 "배경 글리프 1장 + 타이틀 문구 + 아이콘 글리프 목록"만 목록화.
- UI 간 이동(뒤로가기·상위메뉴 복귀) 공통 패턴 1안.

### 2.7 비범위 (1차)
- 실제 GUI 배경 이미지 제작.
- 음수 스페이스 픽셀 단위 튜닝.
- 웹 관리자 UI, 디스코드 봇 임베드 (별도 트랙).

### 2.8 구현 포인트
- 글리프 매핑 JSON을 한 파일(`font/default.json`)로 일원화해 코드가 문자 1개만 붙이면 되게 한다.
- 플러그인 측은 "이 UI는 이 글리프 코드"만 알고, 좌표 계산은 음수 스페이스 유틸로 추상화.

## 3. 무기 (3D 모델·텍스처)

### 3.1 개요
전사 3무기(대검·태도·창) T1/T2 세트를 CustomModelData 기반 아이템 모델로 구현한다. 기능(각인·세트효과·스탯)은 기존 축을 변경하지 않는다.

### 3.2 목표
- 각 무기종 × 세트 × 티어마다 고유 외형을 부여한다.
- 기능은 같고 외형만 다른 "염색/변형" 체계를 별도 레이어로 설계해, 치장 경제와 기능 경제를 분리한다.

### 3.3 핵심 규칙 (무기 CustomModelData 할당표 초안)
- 5세트를 `S1~S5`로 표기. 세트 구분은 마지막 `EE` 연번으로 표현.
- 예약표 (장비·무기, A=1, B=1=전사):
  - 1111000~1111099 = T1 대검 (연번 00~19는 S1, 20~39는 S2 …)
  - 1112000~1112099 = T1 태도
  - 1113000~1113099 = T1 창
  - 1121000~1121099 = T2 대검
  - 1122000~1122099 = T2 태도
  - 1123000~1123099 = T2 창
- **파츠 수 (초안)**: 무기종당 3D 모델 1개 + 텍스처 1장(T1 기본) / T2는 세트별 1~2개 변형 (총 5~10개 변형 사이에서 세트 정체성 유지). 1차는 **세트별 1 변형**으로 최소화.
- **치장 변형 레이어**: 같은 `item_id + 기능` 위에 `cosmeticSkinId`를 별도 NBT로 올려, 장착 시 리소스팩이 다른 모델을 참조하게 한다. 치장 변형은 `AB=20~29` 치장 네임스페이스를 사용하되, 기능 아이템이 참조하는 구조.

### 3.4 1차 범위
- 전사 3무기 × T1/T2 × 5세트 = 총 30 슬롯의 **번호 예약만** 확정.
- 실제 모델은 **T1 대검 기본형 1개**로 프로토타입 → 파이프라인 검증 후 확장.
- 치장 변형 레이어 개념 명세(기능-외형 분리)만 문서화.

### 3.5 2차 이후
- 태도/창 T1 프로토타입 확장.
- T2 세트별 외형 차별화 2~3세트 우선 제작.
- 염색 변형(색상 팔레트 치환) 시스템.

### 3.6 비범위 (1차)
- 마법사/궁수/도적 등 타 직업 무기 (추후 직업 추가 시점).
- 애니메이션(무기 휘두름 모션) — Paper 기본 모션 사용.
- 방어구 3D 모델(1차는 외형 슬롯 없음 가정).

### 3.7 구현 포인트
- 모델 파일은 `models/item/weapon/<weapon_code>/<variant>.json`.
- 아이템 판정(공격력·세트효과)은 기존 EmpireRPG 서비스가 담당하고, 리소스팩은 "보여주기"만 한다.

## 4. 치장 아이템 (꾸미기)

### 4.1 개요
기능 성장과 분리된 외형 전용 슬롯을 신설해, 의상·망토·등 장식·탈것 외형을 운용한다. 이모트는 1차 비범위.

### 4.2 목표
- 기능(능력치)과 외형을 완전히 분리해 시즌 경제에 부담을 주지 않는다.
- 치장 획득 경로를 다각화하되 거래/귀속 원칙을 명확히 해 경매 시장을 보호한다.

### 4.3 핵심 규칙 (2026-04-19 갱신)
- **외형 슬롯 (1차)**: 무기 외형 / 망토(등 장식) / 탈것 외형 3슬롯. 의상(풀셋 방어구)은 1차 비범위.
- **획득 경로 = 랜덤박스 + 확정 구매 병행 (2026-04-19 확정)**: 랜덤박스 단독 아닌 **양방향**. 확정 구매 시 원하는 치장을 고정 가격으로 구매, 랜덤박스는 부가 재미·수집 요소. 이벤트 한정 드랍·시즌 보상·수도 NPC 판매 등 기존 경로도 유지.
- **치장 아이템 귀속 = X 확정 (2026-04-19)**: 모든 치장 아이템 경매 거래 가능(귀속 분류 폐기). 업적·칭호 부속물·시즌 랭커 보상 등 특수 치장도 경매 가능. 단 **치장 블록**(수도 NPC 판매, 건축·배치용)은 별개 축 — 치장 블록의 거래 규칙은 상점 NPC 판매 원칙 따름.
- **등급**: 일반 / 고급 / 희귀 / 영웅 / 전설 5단계(아이템 등급 축 재사용, 별도 등급 체계 신설 금지).
- **CustomModelData 예약**: 대분류 `A=2`, 세부는 `B=0`(공용) 기본. 2021000~ 영역을 치장·무기 변형에, 2022000~을 망토, 2023000~을 탈것 외형에 예약.

### 4.4 1차 범위
- 외형 슬롯 3종(무기·망토·탈것) 개념 확정.
- 획득 경로 2분류(경매 가능 / 계정 귀속) 원칙 확정.
- CustomModelData 예약 구간 확정.
- 수도 상점 NPC 치장 블록 판매와의 호환 재확인: 치장 블록은 **건축/배치용**, 치장 아이템은 **캐릭터 외형용**으로 축을 분리해 UI/획득 경로를 섞지 않는다.

### 4.5 2차 이후
- 의상(풀셋 외형) 슬롯.
- 이모트(동작) 시스템.
- 칭호+배지+치장 조합 프로필 프리셋 저장.

### 4.6 비범위 (1차) — 2026-04-19 갱신
- ~~치장 랜덤박스/가챠~~ → **1차 범위로 승격 (랜덤박스 + 확정 구매 병행 확정)**. 랜덤박스 확률표·피티 보상·확정 구매 시세표 설계는 후속(feature-planner).
- 치장 거래 수수료·세금 (경매장 공통 규칙 따름).
- 치장 염색·색상 커스터마이즈.

### 4.7 구현 포인트
- 치장 장착은 별도 "외형 슬롯 전용 GUI"에서 처리하고, 실제 장비 슬롯과 분리한다.
