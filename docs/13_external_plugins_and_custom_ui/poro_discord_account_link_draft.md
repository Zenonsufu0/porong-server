# 마인크래프트 ↔ 디스코드 계정 자동 연동 실현 가능성 검토

## 1. 구현 목표

포로 서버 플레이어의 마인크래프트 계정과 디스코드 계정을 1:1로 연결하여, 차원도서관(인게임)과 디스코드 봇(외부)에서 동일한 플레이어 식별자로 공개 정보를 조회·공지할 수 있게 한다. "자동"은 제로 클릭이 아니라, **최초 1회 인증 → 이후 재접속 시 무인증 인식**을 의미하는 실용적 범위로 정의한다.

### "자동"의 의미 범위 정의
- 제로 클릭 연동은 **기술적으로 불가능**하다. 마인크래프트(Mojang/Microsoft)와 디스코드는 별개 ID 체계이며, 두 시스템 간 신뢰 경로가 존재하지 않는다.
- 따라서 최초 1회는 반드시 양쪽 계정을 모두 소유한 사람만 통과할 수 있는 **챌린지-응답 인증**이 필요하다.
- 최초 링크 이후 `player_discord_link` 테이블 매핑을 기반으로 재접속/재로그인 시 자동 인식한다. 이것이 포로 서버에서 말하는 "자동"의 실질적 범위다.

## 2. 선행조건

- 약관 v1 동의 시스템(`agreement_version_master`, `player_agreement_log`)이 운영 중이고, "공개 동의" 항목이 디스코드 연동 사전 조건으로 연결 가능해야 한다.
- 디스코드 봇 계정 발급, 봇 토큰 저장소(환경변수 또는 KMS), 포로 전용 길드 ID 확정.
- SQLite write-through cache 쪽에 `player_discord_link` 테이블 추가 가능.
- 마인크래프트 플러그인 측에서 봇과 통신할 수 있는 방식 확정: 동일 프로세스(JDA를 Paper 플러그인에 내장) 또는 별도 프로세스 + 내부 HTTP/메시지 채널.

## 3. 가능한 연동 방식 비교

### (A) 인게임 → 디스코드 OTP 교환
플레이어가 인게임에서 `/discord link`를 입력하면 서버가 6자리 OTP를 생성하여 채팅창에 띄운다. 플레이어가 포로 길드 봇에 DM 또는 슬래시 명령어 `/verify <OTP>`를 전송하면 봇이 OTP를 검증하여 링크를 확정한다.
- **장점**: 웹 인프라 불필요, 구현 단순, 포로 서버 현재 스택에 가장 잘 맞음.
- **단점**: 플레이어가 길드에 가입되어 있고 봇을 찾을 줄 알아야 한다. OTP 유효시간·재발급 정책 필요.
- **UX**: 중간 수준. 두 창을 왔다갔다 해야 하지만 마크 커뮤니티에 익숙한 패턴.

### (B) OAuth2 웹 리다이렉트
인게임에서 단축 URL을 띄우고, 웹 페이지에서 디스코드 OAuth2로 로그인하면 서버가 콜백으로 `discord_user_id`를 받아 진행 중인 세션과 매칭한다.
- **장점**: 가장 공식적, 디스코드 ID·아바타·닉네임을 안전하게 획득.
- **단점**: HTTPS 엔드포인트, 도메인, 리버스 프록시, 세션 스토어, OAuth 앱 설정이 필요하다. 포로 서버 1차 범위에 비해 인프라 부담이 크다.
- **UX**: 상위. 단, 세션-플레이어 매칭을 위한 별도 단기 토큰이 여전히 필요하다(즉 내부적으로는 OTP와 유사).

### (C) 디스코드 → 인게임 확인 프롬프트
플레이어가 봇에 `/link <minecraft_name>`을 입력하면 서버가 해당 플레이어가 온라인일 때 채팅 또는 북 UI로 확인 프롬프트를 띄운다. 플레이어가 "수락"을 누르면 링크 확정.
- **장점**: 플레이어 관점에서 가장 "자동"에 가까운 느낌.
- **단점**: `minecraft_name`을 자칭할 수 있으므로 확인 프롬프트가 반드시 인게임에서 이루어져야 한다. 플레이어가 오프라인이면 대기 큐 처리 필요.
- **UX**: 상위. 단 구현 복잡도는 (A)보다 높다.

### (D) 추가 안: 이중 플로우(A+C 결합)
어느 쪽에서 시작해도 반대쪽에서 확인받는 챌린지-응답을 공통 구현. 장기적으로 가장 유연하지만 1차 버전에는 과하다.

## 4. 추천안

**(A) 인게임 OTP + 디스코드 슬래시 명령어** 방식을 1차 버전으로 채택한다. 이유는 다음과 같다.

- 약관 동의 플로우와 자연스럽게 연결된다. 플레이어가 인게임 최초 입장 시 약관 v1 동의 UI에서 "공개 동의" 선택 직후 `/discord link` 안내를 띄울 수 있다.
- 웹 인프라 없이 JDA 하나만 추가하면 된다. 기존 Paper 플러그인 스택과 충돌이 적다.
- 차후 (B) OAuth 방식으로 확장해도 내부 OTP 교환 로직을 그대로 재사용할 수 있다.

## 5. DB 스키마 추가 제안

`player_discord_link` 테이블 컬럼:
- `player_uuid` TEXT PRIMARY KEY
- `discord_user_id` TEXT NOT NULL UNIQUE (동일 디스코드 계정으로 여러 마크 계정 연동 차단)
- `linked_at` TIMESTAMP
- `link_method` TEXT (`otp_ingame`, `oauth_web`, `bot_prompt`, `admin_manual`)
- `is_verified` BOOLEAN (초기 false, 챌린지 통과 시 true)
- `unlinked_at` TIMESTAMP NULL (해제 이력 보존용, soft delete)
- `agreement_version` TEXT (링크 당시 동의한 약관 버전 스냅샷)

보조 테이블 `player_discord_link_otp`: `player_uuid`, `otp_hash`, `issued_at`, `expires_at`, `attempt_count`. OTP는 평문 저장하지 않고 해시 저장.

## 6. 디스코드 봇 권한 범위

1차 버전 필요 OAuth Scope:
- `bot` (슬래시 명령어 등록 및 DM 발송)
- `applications.commands` (슬래시 명령어)
- `identify` (링크 시 디스코드 유저 ID, 글로벌 네임 획득)

선택적 Scope(도입 시점 별도 검토):
- `guilds.join` (자동 길드 초대) — 사용자 체감 좋지만 동의 UX가 무거워짐. 2차 이후.
- `guilds.members.read` (길드 내 역할 조회) — 랭킹/역할 자동 부여 기능 도입 시.

봇 길드 권한: 지정 공지 채널에 메시지 전송, 역할 부여(선택), 슬래시 명령 등록.

## 7. 작업 분해 (PR 단위)

1. **PR1 스키마·마이그레이션**: `player_discord_link`, `player_discord_link_otp` 테이블 추가, write-through cache 연동.
2. **PR2 디스코드 봇 부트스트랩**: JDA 의존성 추가, 봇 토큰 환경변수 로딩, 핑 명령어까지만.
3. **PR3 인게임 OTP 발급 명령어**: `/discord link`, OTP 해시 저장, 만료 10분, 재발급 쿨다운.
4. **PR4 봇 `/verify` 슬래시 명령어**: OTP 검증, `is_verified=true` 확정, 동일 `discord_user_id` 중복 차단.
5. **PR5 약관 동의 플로우 연결**: 최초 입장 UI에서 공개 동의 직후 링크 안내, `agreement_version` 스냅샷 저장.
6. **PR6 링크 조회·해제**: `/discord status`, `/discord unlink`, 봇 측 동일 기능. 해제는 soft delete.
7. **PR7 공지 연동 1차**: 보스 킬/경매 낙찰 공지에서 링크된 디스코드 멘션 사용.

## 8. 기술 리스크

- **스푸핑**: (C) 방식 단독 사용 시 타인 닉네임 자칭 위험. OTP 방식에서도 OTP가 채팅 로그로 노출될 수 있으므로 **반드시 플레이어 본인에게만 보이는 메시지(예: `sendMessage` 개인 전송)** 로 띄운다.
- **봇 토큰 유출**: 환경변수/시크릿 매니저로만 로드. 리포지토리·로그·크래시 리포트에 포함되지 않게 필터. 유출 시 토큰 즉시 재발급 런북 필요.
- **동일 디스코드로 여러 마크 연동**: `discord_user_id` UNIQUE 제약 + 링크 시 기존 링크 존재 여부 사전 검사. 멀티 계정 허용 정책은 운영 결정 사항(오픈 질문).
- **레이스 컨디션**: 동일 OTP에 대한 동시 검증, 동일 디스코드 ID 동시 링크 시도. DB 제약 + 트랜잭션으로 방어.
- **봇 다운타임**: 봇 장애 시 인게임 진행은 막지 않는다. 링크는 "선택" 단계로 취급한다.
- **디스코드 ID 변경·계정 삭제**: discord_user_id 자체는 영구적이나 글로벌 네임은 변한다. 캐시된 닉네임은 주기적 갱신.

## 9. 테스트 포인트

- OTP 만료·재발급 쿨다운·시도 횟수 제한 단위 테스트.
- 동일 `discord_user_id`로 두 번째 마크 계정 링크 시도 거부 확인.
- soft delete 후 재링크 시 이력 보존 확인.
- 약관 공개 동의를 하지 않은 플레이어가 `/discord link`를 호출했을 때 차단되는지.
- 봇 오프라인 상태에서 `/discord link` 호출 시 인게임 안내 메시지가 정상 출력되는지.
- 마크 서버 강제 종료 후 진행 중 OTP가 만료되는지.

## 10. 1차 버전 권장 범위

- (A) OTP 방식 단독 구현.
- `player_discord_link`, `player_discord_link_otp` 스키마와 기본 CRUD.
- 인게임 `/discord link`, `/discord status`, `/discord unlink`.
- 봇 `/verify`, `/status`, `/unlink`.
- 약관 공개 동의와 연결된 안내 UX.
- 공지 연동은 "보스 킬 공지 시 멘션" 정도의 최소 범위.

## 11. 나중으로 미뤄도 되는 것

- OAuth2 웹 리다이렉트 (B): 웹 포털이 생길 때 함께 도입.
- 디스코드 시작 (C) 방식 확인 프롬프트.
- `guilds.join`을 이용한 자동 길드 초대.
- 디스코드 역할 자동 부여(시즌 클리어·랭커·장인 역할 등).
- 차원도서관에서 디스코드 아바타 렌더링.

## 12. 시즌 전환·계정 이전 시 링크 처리 방침

- **시즌 전환**: 시즌 단위로 진행도는 초기화되지만 `player_discord_link`는 **영구 테이블**이다. 시즌 전환 시 링크는 유지, 단 공개 동의 버전이 달라지면 재동의 요구.
- **계정 이전(마크 계정 변경)**: 관리자 수동 절차 전용. `admin_manual` 메서드로 이전 UUID의 링크를 `unlinked_at` 처리하고 새 UUID에 동일 `discord_user_id`로 재링크. 자동화는 1차 범위 외.
- **디스코드 계정 변경(탈퇴·재가입)**: 플레이어가 `/discord unlink` 후 신규 계정으로 재링크. 기존 공지/이력은 UUID 기준이므로 유지된다.

## 13. 오픈 질문

- 한 디스코드 계정이 여러 마크 계정을 링크할 수 있게 허용할 것인가, 아니면 1:1 강제인가. (현재 제안: 1:1 강제.)
- 링크하지 않은 플레이어의 차원도서관 공개 정보는 어디까지 노출하는가. 디스코드 멘션만 제한하는가, 아예 조회도 제한하는가.
- 봇을 Paper 플러그인 내장(동일 프로세스)으로 갈 것인가, 별도 프로세스로 분리할 것인가. 운영 안정성과 배포 편의성 사이 선택.
- 약관 v1 공개 동의 거부 시 디스코드 링크 자체를 막을 것인가, 링크는 허용하되 공개 기능만 막을 것인가.
- OTP를 6자리 숫자로 할지 8자리 영숫자로 할지. (현재 제안: 6자리 숫자, 10분 만료.)
