## 기존 포로 서버 기준과의 충돌 점검

CLAUDE.md 원칙에 따라 본 검토가 기존 확정 기준과 충돌할 수 있는 지점을 먼저 정리한다.

- **리소스팩 도입 확정 (2026-04-19)**: 3단계 배포 정책(자동 → 디스코드 수동 → 체험 월드 강등), 7자리 CustomModelData 인코딩, 공용 글리프 7종 + UI 7종 모두 확정. 본 문서는 이 축을 그대로 유지한다.
- **HUD 3레이어 (2026-04-19 I 권장안 전체 수용)**: Adventure `sendActionBar()` 단일 멀티라인 + font glyph + 바닐라 HP/허기 2중 방어. 본 문서는 이 구조의 **제작 실측**만 다루며 구조를 뒤집지 않는다.
- **CustomModelData CSV 레지스트리 (PR#1~#4 구현됨)**: 13 컬럼 + R1~R5 린트 이미 존재. 본 문서는 예약 구간 사용량 추정에만 관여.
- **맵 제작 1인 결정 / 외주 없음 / 외부 도구 미도입**: 이전 사용자 결정. 본 문서도 같은 맥락에서 "1인 제작 기본값"으로 옵션을 정렬한다.
- **기존 `HealthHudFormatter`·`HealthHudListener`**: HUD 3레이어 문서에서 "폴백 텍스트 렌더러로 역할 축소" 방향 합의. 본 문서는 해당 방향을 그대로 전제한다.

충돌 지점 없음. 단 **1인 픽셀 아트 제작 실측 공수**와 **음수 스페이스 수직 배치 기술 검증**은 사용자 확인 없이 확정하지 않는다(§9 오픈 질문).

## 1. 기능 개요

포로 서버 전용 커스텀 UI(리소스팩 font glyph + Adventure 컴포넌트 기반)의 실제 제작 가능성을 "1차 공개 기준"으로 정리한다. 이미 확정된 설계를 다시 그리지 않고, **"지금 가진 인력/시간/기술로 얼마나 제작 가능한가"**만 실측 관점으로 본다.

범위: 7 UI(올인원 조합대 · 월렛 · 인벤토리 보조 · 영지 · 경매 · 퀘스트 저널 · Shortcut Menu) + HUD 3레이어.

---

## 2. 왜 필요한가

- 리소스팩 설계는 확정됐지만 **실제 에셋 제작 주체·공수**는 아직 실측된 적이 없다.
- 7 UI를 한 번에 만들면 시즌1 오픈이 지연될 가능성이 크다. 1차에 무엇을 최소로 내보내고 무엇을 2차로 미룰지 "공개 기준선"이 필요하다.
- 픽셀 아트 외주 여부, 공개 에셋 활용 여부는 비용·품질·기간 트레이드오프가 크다. 결정 근거를 문서로 남긴다.
- HUD 3레이어는 PR 7~8개, 10~13 man-day로 분해돼 있으나 리소스팩 에셋 쪽 공수는 별도 트랙이라 통합 로드맵이 빠져 있다.

---

## 3. 핵심 사용자 경험 (제작 과정 관점)

- **기획자(사용자) 관점**: UI별 에셋 목록·글리프 수를 먼저 확정 → 제작 우선순위대로 작업 → 완료된 UI부터 CSV 등록 + 리소스팩 반영 → QA.
- **플레이어 관점 (폴백 경로 정합)**:
  - 리소스팩 적용: 풀 커스텀 UI 체험.
  - 리소스팩 미적용 → 디스코드 수동 복구 → 재접속.
  - 끝까지 미적용: **체험 월드 강등**. 본 검토의 7 UI는 체험 월드에서 **기본 GUI 타이틀 + 텍스트 폴백**으로 대체됨(HUD 3레이어 §6 비범위 규약과 동일).

## 4. 핵심 규칙

### 4.1 UI별 공수 단위 정의 (실측 요건 범주)

각 UI는 아래 5개 공수 단위를 가진다. 본 문서는 **단위 정의와 상대 비교**만 하며, 절대 공수는 사용자 확인 필요.

- **a. 배경 글리프 1장**: UI 창 배경 이미지(픽셀 아트). 해상도는 폰트 글리프 표준(권장 256×128 내외, 음수 스페이스 튜닝 전제).
- **b. 아이콘 글리프 N종**: 버튼·탭·상태 아이콘. 공용 글리프 7종은 재사용.
- **c. 클릭 영역·아이템 배치**: 바닐라 인벤토리 슬롯 기반 + 커스텀 아이템 매핑(Paper Inventory API).
- **d. 이벤트 리스너 구현**: EmpireRPG 플러그인 내 `InventoryClickEvent` 분기 + 상태 갱신.
- **e. 상호작용 복잡도**: 정적 표시만 vs 동적 갱신(페이지·검색·필터·실시간 시세) 여부.

### 4.2 난이도 등급 (상대 비교용)

- **Lv.1 정적 단일 화면**: 배경 1 + 아이콘 3~5 + 고정 표시. (예: 월렛, Shortcut Menu)
- **Lv.2 다중 탭 + 페이지**: 배경 2~3 + 아이콘 10 내외 + 페이지 넘김. (예: 인벤토리 보조, 퀘스트 저널)
- **Lv.3 동적 갱신 + 검색·필터**: 배경 3~5 + 아이콘 15+ + 실시간 DB 조회. (예: 올인원 조합대, 영지, 경매)

### 4.3 제작 주체 원칙 (사용자 확인 필요)

- **픽셀 아트**: 1인 서버 원칙상 "본인 제작" 기본값. 외주·유료 에셋·무료 공개 에셋 활용 여부는 사용자 결정.
- **Java 이벤트 리스너**: EmpireRPG 코드베이스 내 자체 구현 가능(기존 `HealthHudListener` 패턴 재사용).
- **리소스팩 빌드·배포**: 이미 3단계 배포 흐름 설계 존재, 추가 인력 없음.

## 5. 7 UI 제작 난이도 · 우선순위 표

> 숫자는 **상대 규모 추정**이며 절대 공수가 아니다. 실제 시간은 §7 옵션 비교 참조.

| UI | 난이도 Lv | 배경 장수 | 아이콘 글리프 수 | 상호작용 | 시즌1 오픈 필수 | 폴백 경로 |
|---|---|---|---|---|---|---|
| **올인원 조합대** (WB01) | Lv.3 | 3~5 | 15~20 | 동적(레시피 검색·조합 처리) | **필수** — 제작 시스템 핵심 | 바닐라 조합대 차단 기조상 폴백 제한적, 체험 월드로 강등 유도 |
| **월렛** (WA01) | Lv.1 | 1 | 3~5 | 정적(잔액·화폐 목록 표시) | **필수** — 화폐 조회 기본 | 텍스트 채팅 명령 `/empire wallet` 대체 가능 |
| **인벤토리 보조** (IV01) | Lv.2 | 2~3 | 8~10 | 반동적(장비 슬롯 프리뷰) | 권장(2차 이월 가능) | 기본 인벤토리 GUI로 충분 |
| **영지** (ES01) | Lv.3 | 3~4 | 15~20 | 동적(영지 상태·자원·멤버) | 2차 이월 가능(영지 시스템 개방 시점에 맞춤) | 채팅 명령 대체 가능 |
| **경매** (AU01) | Lv.3 | 3~5 | 15~20 | 동적(실시간 시세·검색·필터) | 2차 이월 권장(시스템 복잡도·시세 안정화 필요) | 디스코드 봇 임베드 조회 대체 가능 |
| **퀘스트 저널** (QJ01) | Lv.2 | 2~3 | 10~12 | 반동적(진행도·목록) | **필수** — 퀘스트 플러그인 편입 맥락 | 기본 인벤토리 GUI로 보조 가능 |
| **Shortcut Menu** (SM01) | Lv.1 | 1 | 5~7 | 정적(Shift+좌클릭 허브) | **필수** — 키 축 확정 맥락 | 채팅 명령 대체 가능하나 UX 저하 큼 |

**1차 공개 필수 추정**: 올인원 조합대 + 월렛 + 퀘스트 저널 + Shortcut Menu = **4종**.
**2차 이월 권장**: 인벤토리 보조 + 영지 + 경매 = 3종.

필수 4종 구성 근거:
- 올인원 조합대: 제작 시스템 전면 차단 후 유일 제작 경로.
- 월렛: 기본 경제 조회 UI.
- 퀘스트 저널: 퀘스트 플러그인 커스텀 편입(04c1635 커밋 맥락).
- Shortcut Menu: Shift+좌클릭 키 축 확정, 대체 UX 없음.

## 6. HUD 3레이어 제작 실측

### 6.1 기술 항목별 가능성

- **`sendActionBar()` 단일 멀티라인 + 음수 스페이스 수직 배치**
  - **상태**: **기술 스파이크 미완료** (HUD 3레이어 §10 다음 작업 #2 "Paper 1.21.10 + Adventure에서 3줄 수직 배치가 한 호출로 안정 렌더되는지 1일 스파이크" 명시). 본 1차 공개 가능성 판단 전 **반드시 선행**.
  - **리스크**: 음수 스페이스 픽셀 튜닝이 Minecraft 클라이언트 버전별로 차이가 있을 수 있다. 실측 전 확정 금지.

- **바닐라 HP/허기 제거 (`setHealthScale(0)` + `setFoodLevel(20)`)**
  - **상태**: Paper 공식 API에 존재, 포로 서버에서 사용 전례 미확인 → **호환성 스파이크 필요**.
  - **2중 방어**: 서버측 호출 + 리소스팩 글리프 투명화 병행 → 한쪽 실패 시 다른 쪽 커버.
  - **리스크**: Citizens / MythicMobs / 기타 HP 렌더 플러그인과의 충돌(HUD 3레이어 §9 오픈 질문 그대로).

- **기존 `HealthHudFormatter` / `HealthHudListener` 전환**
  - **현 구조**: `HealthHudFormatter`는 하트 + 현재/최대 수치 + 10칸 █/□ 블록 텍스트를 생성.
  - **전환 비용**: **낮음**. 폴백 텍스트 렌더러로 역할 축소 합의(HUD 3레이어 §0, §7 선행조건). 신규 `HudRenderService`가 상위에서 리소스팩 미적용/체험 월드 플레이어에만 기존 포매터 호출.
  - **코드 변경**: 기존 수치 포맷 함수 `formatValue` 재사용, 신규 모듈은 별도 패키지 `com.poro.empire.hud.*`.

- **CustomModelData `3002000~3002999` 예약 구간 사용량 추정**
  - 레이어 1 HP 바: 약 30개(배경 4 + 칸 10 + 수치 예비 10 + α).
  - 레이어 2 스킬 바: 약 50개(배경 1 + 슬롯 5 + 원형 8 + 직업/무기별 아이콘 30+).
  - 레이어 3 버프/디버프: 약 100~150개(프레임 5 + 숫자 10 + 개별 아이콘 = 상태 registry 수에 비례).
  - **합계 추정**: 200~250개 / 1000개 구간. **여유 충분**. 공통 오프셋 보정 예비 포함 시에도 30% 이상 여유.

### 6.2 HUD 제작 1차 필수 최소 에셋

- 레이어 1 HP 바 색상 4종 글리프 (full / mid / low / critical).
- 레이어 2 슬롯 테두리 5종 + 원형 쿨다운 8단계 + 직업·무기별 스킬 아이콘 (전사 3무기 × 5슬롯 = 15종 우선).
- 레이어 3 카테고리 테두리 4종 + 숫자 0~9 + 상태 registry 핵심 20~30종.

**필수 총합**: 약 **75~95 글리프**. 1인 픽셀 아트 공수 실측은 사용자 확인 필요.

## 7. 옵션 비교

> 각 옵션의 "기간" 수치는 일반적 1인 제작 시나리오를 가정한 **추정**이며, **사용자 확인 필요**(본인 제작 속도, 가용 시간, 픽셀 아트 경험치에 따라 크게 달라진다).

### 옵션 A — 전부 자체 제작 (풀 커스텀)
- **범위**: 7 UI + HUD 3레이어 모두 자체 픽셀 아트.
- **추정 기간**: 3~6개월 (사용자 확인 필요).
- **장점**: 포로 정체성 최대화, 외부 의존 없음.
- **위험**: 시즌1 오픈 지연 가능성 큼. 1인 제작 피로도 누적.

### 옵션 B — 공개 에셋 + 일부 자체
- **범위**: 공용 글리프 7종·버튼·화살표·숫자 등은 공개 CC0/MIT 에셋 활용, UI 배경과 아이덴티티 요소만 자체 제작.
- **추정 기간**: 2~3개월 (사용자 확인 필요).
- **장점**: 기간 단축, 완성도 중간 확보.
- **위험**: 공개 에셋 라이선스 확인·스타일 이질감 정리 필요.

### 옵션 C — 1차 공개 최소 UI만 (4종 + HUD)
- **범위**: 올인원 조합대 + 월렛 + 퀘스트 저널 + Shortcut Menu + HUD 3레이어 핵심. 나머지 3 UI는 2차 이월.
- **추정 기간**: 1~2개월 (사용자 확인 필요).
- **장점**: 시즌1 오픈 일정 확보, 2차 이월 UI는 플레이어 피드백 후 제작.
- **위험**: 영지·경매·인벤토리 보조는 체험 월드 강등 UX 또는 채팅/봇 폴백에 의존.

### 옵션 D — 리소스팩 미도입, Adventure Component 텍스트만
- **범위**: 기존 `HealthHudFormatter` 확장 + 채팅/인벤토리 타이틀 텍스트 UI만.
- **추정 기간**: 즉시~2주 (사용자 확인 필요).
- **장점**: 즉시 가능, 제작 리스크 최소.
- **위험**: 리소스팩 도입 **확정 결정을 뒤집는 것** — CLAUDE.md 금지 원칙(확정 기준을 이유 없이 뒤집지 않음)에 저촉. 옵션 D는 "비상시 회귀 옵션"으로만 기록하고 1차 후보에서 제외 권고.

## 8. 권장안 + 로드맵

### 8.1 권장 옵션 = **옵션 C (1차 최소 4 UI + HUD)**

근거:
- 옵션 A는 1인 제작 기간 리스크가 지나치게 크다.
- 옵션 B는 기간은 단축되나 "스타일 이질감 정리" 공수가 불확실해 옵션 C보다 리스크가 크다.
- 옵션 D는 확정된 리소스팩 결정을 뒤집어 CLAUDE.md 금지에 해당.
- 옵션 C는 이미 HUD 3레이어 10~13 man-day 공수 추정이 존재하고, 1차 UI 4종이 모두 Lv.1~Lv.3 중 하한에 가깝다.

### 8.2 단계별 로드맵

**1차 공개 (시즌1 오픈)**

- Shortcut Menu (SM01) — Lv.1, Shift+좌클릭 축.

**2차 확장 (30~60일차)**

## 13. 오픈 질문

- **1인 픽셀 아트 실측 공수** — 배경 1장·아이콘 1종 기준 실소요 시간. 사용자 확인 필요.
- **외주 / 유료 에셋 / 공개 에셋 활용 방침** — "외주 없음" 원칙이 유지되는지, 공개 에셋은 어디까지 허용할지.
- **음수 스페이스 수직 배치** — Paper 1.21.10 + 클라이언트 최신판에서 3줄 배치가 깜빡임 없이 되는지.
- **`setHealthScale(0)`** — Citizens·기타 플러그인과의 충돌 여부.
- **2차 이월 UI 폴백 UX** — 영지·경매·인벤토리 보조의 채팅·디스코드 봇 폴백을 1차 시즌 내내 유지해도 경험이 납득 가능한가.
- **HUD 3레이어 에셋 제작 트랙과 7 UI 에셋 제작 트랙의 병렬 진행** — 1인 기준 동시 진행 가능한가, 아니면 순차적으로 해야 하는가.
- **공수 실측 이전에 옵션 C 확정하는 것이 맞는가** — 실측 후 옵션 B로 이동할 여지를 열어둘지.
- **치장·무기 외형 모델 제작**이 본 UI 트랙과 같은 1인 에셋 파이프라인을 점유하는가 — 리소스팩 드래프트 §3 치장/무기 모델은 별도 공수.
