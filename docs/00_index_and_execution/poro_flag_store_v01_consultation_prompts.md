# 플레이어 플래그 저장소 v0.1 오픈 질문 자문 요청 통합 프롬프트 묶음

## 문서 목적
`poro_flag_store_v01_sprint_plan.md` 하단 오픈 질문 4건을 **사용자 직접 확정 대신 implementation-reviewer / codex MCP 추천 방식**으로 처리하기 위해, 각 질문별 자문 요청 프롬프트를 한 묶음으로 정리. 2026-04-19 사용자 결정에 따라 작성.

## 사용 방식

### implementation-reviewer 에이전트 호출 시
- 본 문서 전체를 프롬프트 컨텍스트로 첨부.
- 에이전트에 "아래 4건 중 N번 질문에 대해 응답 포맷대로 답해달라"고 지시.
- 스프린트 플랜 원문(`poro_flag_store_v01_sprint_plan.md`)도 함께 첨부 권장.

### codex MCP 프롬프트로 붙여넣기
- 각 질문 블록을 개별 프롬프트로 분리해 codex에 송신.
- "제약 조건" 섹션을 반드시 포함해서 포로 서버 기준과 충돌 지점을 codex가 먼저 설명하도록 유도.
- codex 응답은 그대로 복사 금지. 포로 서버 기준에 맞게 재정리해야 함(CLAUDE.md 원칙).

### 응답 반영 경로
1. 자문 결과를 받아 포로 서버 기준 재정리(추측 확정 금지, 충돌 지점 명시).
2. 결정이 확정되면 `poro_master_planning.md` 변경 로그 섹션에 반영.
3. `poro_open_questions_registry.md`의 플래그 저장소 v0.1 섹션에서 해당 항목 **취소선 처리 + 결정 요약 1줄**.
4. 필요 시 `poro_flag_store_v01_sprint_plan.md` 오픈 질문 섹션에서도 동일 처리.

## 질문 1 — `FoundationContext` SQLite DataSource/ConnectionProvider 실제 주입 상태 (블로킹)

### (a) 질문 본문
플래그 저장소 v0.1 스프린트 플랜 PR3(`PlayerFlagRepository JDBC`)는 `FoundationContext`가 SQLite `DataSource` 혹은 `ConnectionProvider`를 이미 주입받아 제공한다는 전제 위에 설계되어 있습니다. 현재 코드베이스(`custom-plugins/empire-rpg`) 기준으로 실제 주입이 완료됐는지, 아니면 스텁·NoopDataSource 상태로만 존재하는지, 실제 DataSource 주입 PR이 착수됐는지 확인이 필요합니다. 선행 조건 미충족 시 PR3 착수 자체가 블로킹됩니다.

### (b) 자문 받고 싶은 구체 항목
1. `FoundationContext` 현재 구현체에서 `DataSource` / `ConnectionProvider` / `Database`류 필드가 실제 SQLite 풀(HikariCP 등)에 연결돼 있는가, 아니면 인터페이스만 노출한 스텁 상태인가.
2. 스텁 상태라면 실제 DataSource 주입 PR이 별도로 **계획돼 있는지 / 진행 중인지 / 미착수인지**, 관련 브랜치·PR 번호·담당자가 있는지.
3. 만약 PR3 착수 시점까지 주입이 완료되지 않을 경우, PR3에서 임시 `DataSource` 로컬 부트스트랩(테스트용 HikariCP 직결)을 둘지, 아니면 선행 PR 머지 대기로 확정할지 권장안.
4. 주입 대기 동안 PR1(DDL) / PR2(FlagKey 검증기)는 병렬 진행 가능한지(비-DB 영역이므로 논리적으로 독립해야 함) 확인.
5. 주입 완료 시점에 기존 플러그인(Quest/Honor 등)에서 동일 `DataSource`를 이미 쓰고 있다면 커넥션 풀 사이즈·트랜잭션 격리 수준 기본값이 플래그 저장소의 write-through 가정과 충돌하지 않는지.

### (c) 제약 조건·포로 서버 기준
- `poro_master_planning.md`의 운영 목표: "장비/각인/스탯/통계 데이터를 DB에 저장하고 웹/디스코드 봇에서도 조회 가능"이 전제.
- `poro_flag_store_v01_sprint_plan.md` 선행 조건: "SQLite 연결 풀·`DataSource` 주입 경로가 `FoundationContext`에 존재 확인"이 명시돼 있음.
- 스프린트 범위: SQLite 단일 파일 DB 가정, 분산 DB·외부 RDB로 확장하지 않음.
- v0.1은 write-through 모드. 지연 쓰기·배치 커밋은 v0.2 이후.
- 기존 구조를 유지하면서 확장하는 방향 우선(과한 재설계 금지).

### (d) 응답 포맷 요청
- **현재 주입 상태 진단** (3~5줄 요약)
- **PR3 착수 가능 여부 판정**: 가능 / 선행 PR 대기 / 임시 부트스트랩 허용
- **권장 진행 순서 1개** (단일 명확한 경로)
- **리스크 2개** (각 리스크는 완화 방안 1줄 동반)

## 질문 2 — `NoopMigrationEntryPoint` 교체 PR 현재 단계 (블로킹)

### (a) 질문 본문
스프린트 플랜은 `NoopMigrationEntryPoint` 교체 PR을 **D-0 선행 머지 필수**로 지정하고 있습니다. 마이그레이션 러너가 교체되지 않으면 PR1에서 등록한 DDL이 데드코드가 되어 전체 스프린트가 무의미해집니다. 현재 이 교체 PR이 착수 전 / 진행 중 / 리뷰 중 중 어느 단계인지, 머지 예상 시점과 리스크를 확인해야 합니다.

### (b) 자문 받고 싶은 구체 항목
1. `NoopMigrationEntryPoint` 교체 PR이 현재 **착수 전 / 구현 중 / 리뷰 대기 / 리뷰 중 / 머지 대기** 중 어느 단계인가. 관련 브랜치·이슈·PR이 존재하는가.
2. 교체될 마이그레이션 러너 후보(Flyway / Liquibase / 자체 러너)가 확정됐는가. 아직이라면 v0.1 범위 내에서 가장 위험이 낮은 선택지는 무엇인가.
3. 교체 PR 머지까지 지연이 2~3주 이상 예상될 경우, 스프린트 플랜의 비상안("PR1에 `CREATE TABLE IF NOT EXISTS` 직접 실행 스텁 + 교체 후 PR1.5로 스텁 제거")을 **채택할지 / 교체 PR 머지를 강제 대기할지** 권장안.
4. 비상안 채택 시 기술 부채 크기(스텁 제거 PR1.5 공수, 회귀 리스크) 추정치.
5. 교체 PR 머지 책임 분리(플래그 저장소 스프린트와 별개 담당자인지, 동일 담당자 병행인지)가 v0.1 스프린트 10~12 man-day 산정에 영향을 주는가.

### (c) 제약 조건·포로 서버 기준
- `poro_flag_store_v01_sprint_plan.md` 선행 조건: "`NoopMigrationEntryPoint` 교체 PR = D-0 선행 머지 필수".
- 스프린트 플랜 리스크 섹션: 비상안은 이미 제시돼 있음 (PR1.5 스텁 제거 경로).
- 포로 서버는 작은 단위 설계·기존 구조 유지를 우선. 마이그레이션 러너 교체 자체는 본 스프린트 범위 **밖**(선행 조건이지 산출물 아님).
- 교체 PR 결과는 향후 Quest / Honor / Estate 도메인 DDL에도 공통 사용됨 → v0.1 이후 재활용 자산.

### (d) 응답 포맷 요청
- **현재 단계 진단** (착수 전 / 진행 중 / 리뷰 중 / 머지 대기 중 택 1 + 근거 1~2줄)
- **예상 머지 시점** (불확실하면 "불명 — 담당자 확인 필요"로 명시)
- **권장 결정**: 교체 PR 대기 / 비상안 채택 중 1개
- **리스크 2개** (각각 완화안 1줄)

## 질문 3 — `FlagValueType`에 JSON 허용 여부 (일반)

### (a) 질문 본문
스프린트 플랜은 v0.1 `FlagValueType`을 **BOOL / LONG / STRING 3종**으로 제한하고, JSON은 v0.2로 미루기로 합의돼 있습니다. 그러나 각인 활성화·영지 권한 같은 후속 시스템이 복합 구조를 필요로 할 가능성이 있어, v0.1 단계에서 JSON을 조기 허용하는 것이 비용 대비 실익이 있는지 기술 자문이 필요합니다. "v0.2로 미뤄야 할 이유"가 현재 합의된 근거(마이그레이션 비용 폭증)로 충분한지 재검토 대상.

### (b) 자문 받고 싶은 구체 항목
1. SQLite 기반 `player_flags` 테이블에 `value_json TEXT`(또는 `value_blob`) 컬럼을 1개 추가하는 수준으로 JSON을 조기 허용할 경우, 스키마·Repository·Store·이벤트 계층 각각의 추가 공수 (man-day 단위).
2. JSON 허용 시 실제 쿼리 경로(부분 필드 인덱싱·검색·역직렬화 캐시 등)를 v0.1에서 지원해야 하는가, 아니면 "저장·전체 읽기만" 지원해도 후속 3시스템(수도 반경 / 영지 블록 / 각인 활성화)이 충분히 동작하는가.
3. JSON 허용 시 마이그레이션 비용 폭증 리스크(스프린트 플랜이 지적한)가 구체적으로 어느 지점에서 발생하는가. v0.1→v0.2 전환 시 재변환이 실제 필요한가.
4. BOOL/LONG/STRING 3종만으로 후속 3시스템 요구를 커버 가능한가. 각인 활성화가 "슬롯 인덱스 + 각인서 ID + 활성화 시각" 복합 구조를 필요로 할 경우 STRING 직렬화(CSV·구분자) 우회가 현실적인가, 아니면 JSON이 필수인가.
5. 최종 판단: v0.1 **유지 (3종)** / v0.1에서 **JSON 조기 허용** / v0.1은 유지하되 **v0.2 일정을 명시적으로 당김** 중 권장안.

### (c) 제약 조건·포로 서버 기준
- `poro_flag_store_v01_sprint_plan.md` 스키마 제한: "`FlagValueType` enum: **BOOL / LONG / STRING만** 허용 (JSON은 v0.2)".
- 리스크 섹션: "Value 직렬화 스키마 폭주" 방지 위해 v0.1에서 JSON 배제.
- v0.1 총 공수 10~12 man-day 목표 유지. 범위 확장 금지가 원칙.
- 플래그 저장소는 후속 3시스템(수도 반경·영지 외 블록 금지·각인 활성화) 선행 인프라 — 이들 요구 수준이 최소 커버리지 기준.
- 포로 서버 기준: 기존 구조 유지하며 확장. 과한 재설계 금지.

### (d) 응답 포맷 요청
- **3종 유지 / JSON 조기 허용 / v0.2 일정 당김** 3안 장단점 표 (공수·리스크·후속 시스템 영향 열 포함)
- **권장안 1개** (명확한 단일 결정)
- **리스크 2개** (권장안 채택 시 남는 기술 부채·장래 확장 리스크)

## 질문 4 — 변경 이벤트 방식: Bukkit Event vs 내부 이벤트 버스 (일반)

### (a) 질문 본문
PR6(`PlayerFlagChangedEvent` 발행)은 현재 Bukkit 표준 `Event`로 전제돼 있으나, 내부 이벤트 버스(커스텀 dispatcher)와 어느 쪽이 포로 서버 상황에 더 적합한지 합의가 완료되지 않았습니다. 성능·테스트 용이성·플러그인 간 결합도 관점의 트레이드오프 자문이 필요합니다.

### (b) 자문 받고 싶은 구체 항목
1. Bukkit `Event` 채택 시 장점(표준 리스너 모델·다른 플러그인에서도 구독 가능·`EventPriority` 제공) vs 단점(메인 스레드 강제·async 이벤트 주의·단위 테스트 시 Bukkit 의존 mock 필요).
2. 내부 이벤트 버스(예: Guava EventBus·자체 `FlagEventDispatcher`) 채택 시 장점(Bukkit 분리·테스트 용이·async 제어 자유) vs 단점(외부 플러그인에서 구독 불가·커스텀 에러 처리 필요·학습 곡선).
3. 후속 3시스템(수도 반경 / 영지 외 블록 / 각인 활성화)이 모두 **동일 플러그인 내부**에서 구독할 가능성이 높다면 내부 버스가 오히려 자연스러운지, 아니면 향후 다른 플러그인(디스코드 봇 브리지 등) 확장 가능성을 고려해 Bukkit Event가 더 안전한지.
4. 하이브리드안: **내부 버스가 1차 구독점, 필요 시 Bukkit Event로 브리지 포워드**하는 2단 구조의 복잡도가 v0.1 공수 범위(PR6 0.5 man-day) 안에 들어오는지.
5. `batchSet` API 사용 시 이벤트 폭주(각인 일괄 변경 시 N개 발행) 방지 관점에서 Bukkit Event가 유리한지 내부 버스가 유리한지. v0.2 배치 이벤트 전환 난이도 차이.

### (c) 제약 조건·포로 서버 기준
- `poro_flag_store_v01_sprint_plan.md` PR6: `PlayerFlagChangedEvent(player, flagKey, oldValue, newValue, source)`, Store.set/remove에서 **트랜잭션 성공 후 발행**. 이것은 양쪽 방식 모두 지켜야 하는 공통 규약.
- v0.1 배치 정책: `batchSet` 시 N개 개별 발행 허용 (배치 이벤트는 v0.2).
- 포로 서버 운영 목표: 웹·디스코드 봇에서도 플래그 상태 조회 필요 → 장기적으로는 플러그인 간 경계를 넘나드는 알림 필요성 존재 (단, v0.1 범위는 서버 내부).
- 기존 포로 서버 플러그인이 이미 Bukkit Event를 선호해 쓰고 있는지, 내부 디스패처 패턴 사례가 있는지가 중요한 판단 근거(코드베이스 현황 확인 요청).
- 과도한 추상화·대규모 리팩터링 금지.

### (d) 응답 포맷 요청
- **Bukkit Event / 내부 이벤트 버스 / 하이브리드** 3안 장단점 표 (성능·테스트 용이성·플러그인 간 결합도·v0.1 공수 열 포함)
- **권장안 1개** (단일 결정, 근거 2~3줄)
- **리스크 2개** (선택 시 장래 부담·확장 제약)
