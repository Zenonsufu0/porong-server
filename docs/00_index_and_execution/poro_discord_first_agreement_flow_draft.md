# 디스코드 선행 약관 동의 → 마크 화이트리스트 하이브리드 플로우 (초안)

## 문서 목적
약관 v1 + 디스코드 자동 링크 기존 (A)안을 **디스코드 선행 약관 동의 플로우**로 확장하는 하이브리드 설계. 약관 수집은 디스코드에서, 본인 확인은 마크 OTP에서 분담. implementation-reviewer γ(하이브리드) 권장안 기반.

## 목표

디스코드 서버를 1차 게이트로 삼아 약관 동의와 마크 닉네임 등록을 처리하고, Paper 서버는 화이트리스트에 등록된 유저만 받는 구조가 기존 (A)안 대비 실현 가능한지, 그리고 포로 서버 기준(`player_agreement_log`, `player_discord_link`, v1.1 재동의 플로우)과 충돌 없이 병행할 수 있는지를 판별한다. 최종적으로 α(디스코드 선행 단독) / β(A안 단독) / γ(하이브리드) 중 **γ 권장**.

## 선행조건

- 기존 `agreement_version_master` / `player_agreement_log` 스키마가 이미 준비되어 있다.
- 기존 `player_discord_link`(`discord_user_id` UNIQUE 1:1) 가정은 유지한다.
- Paper 서버 화이트리스트 모드가 ON이며, 봇이 해당 엔트리를 조작할 수 있는 권한 경로가 최소 1개 존재한다.
- 약관 v1 본문이 디스코드 임베드 길이(4096자) 안에 수납되거나, 초과 시 스크롤 가능한 모달 + 외부 링크로 대체 가능해야 한다.
- 포로 운영 정책이 "디스코드 가입이 사실상 필수"라는 전제에 동의해 주어야 한다. 이 전제가 깨지면 α는 자동 탈락.

## 작업 분해

### 1. 기술적 실현 가능성 (2026-04-19 갱신 — 봇 자동화 파이프라인 단일 경로 확정)
- **약관 본문 위치 = 디스코드 봇 임베드 단일 경로 확정**. 외부 정적 링크 보조안은 폐기. 긴 본문은 임베드 다중 페이지(reaction-pagination)로 분할하되 전문은 봇이 직접 노출하며, "아래까지 읽었음" 버튼 클릭 타임스탬프가 읽음 증빙.
- **자동화 파이프라인 순서 확정**: 유저 디스코드 서버 최초 입장 → 봇 GuildMemberAdd 훅이 웰컴 채널/DM에 약관 임베드 자동 발송 → 유저 동의 버튼 클릭 → 봇이 원자적으로 (a) `player_agreement_log` 인서트, (b) 디스코드 "동의자" 역할 부여(서버 내부 채널 접근 권한 개방), (c) `player_whitelist` pending 인서트 3단계 수행. 1단계라도 실패 시 전체 롤백 + 재시도 큐로 이월, 봇 관리자 채널 알림.
- Discord 컴포넌트는 버튼 + 모달(텍스트 인풋 최대 5개)로 충분히 "약관 동의 체크 + 마크 닉네임 입력 + 생년 입력" 조합이 가능하다.
- Mojang API(`api.mojang.com/users/profiles/minecraft/<name>`)는 레이트 리밋이 IP 기준으로 있고, 가끔 429를 돌려준다. 봇에서 60초 캐시 + 실패 시 재시도 1회 정도면 충분. 프리미엄 미가입 이름은 404이므로 즉시 거절 메시지로 안내.
- 화이트리스트 조작 경로는 4개 중 **"봇이 공용 MySQL의 `player_whitelist` 테이블만 쓰고, Paper 측 `PlayerLoginEvent`에서 해당 테이블을 읽어 허용/거절"** 방식이 가장 단순하다. Paper 내장 `whitelist.json`을 봇이 직접 쓰는 방식은 파일 락·재시작 이슈 때문에 피한다. EasyWhitelist/LuckPerms는 외부 의존 증가.
- 봇 프로세스 위치는 **Paper 내장 플러그인 방식은 비추천**한다. 서버 재시작 = 봇 다운이 되어 신규 가입 흐름이 같이 죽기 때문이다. 별도 봇 프로세스 + 공용 DB(기존 SQLite → 필요 시 MySQL 승격)로 느슨하게 연결한다.

### 2. 닉네임 본인 확인 (핵심 리스크)
- 무검증(A)은 유명 유튜버 닉 선점·타인 UUID 선점이 가능해 사실상 채택 불가.
- Mojang OAuth(C)는 MS가 외부 OAuth를 공개하지 않아 사실상 불가능. 검토 종결.
- 현실적인 방식은 **(B) 역방향 OTP 하이브리드**: 디스코드에서 닉네임을 받아 화이트리스트에는 "pending" 상태로 올리고, 실제 해당 UUID가 첫 로그인했을 때 인게임에서 "디스코드에 표시된 6자리 코드를 채팅으로 입력" 또는 "`/confirm <code>`"을 요구. 이때 비로소 링크가 `is_verified=true`로 확정된다.
- 이 방식이면 타인 닉네임을 화이트리스트에 올려도 그 UUID의 실제 소유자가 OTP를 보지 못하므로 탈취가 차단된다. pending이 N시간(예: 24h) 내 확정되지 않으면 엔트리 자동 만료.

### 3. 기존 A안과의 관계 (2026-04-19 갱신 — v1.1 재동의 디스코드 통합)
- 완전 대체는 위험하다. 약관 본문 가시성·본인 확인·봇 다운타임 리스크를 모두 디스코드 한쪽에 몰아넣게 되기 때문이다.
- 합성 가능한 포인트: "약관 본문 표시·동의 수집"은 디스코드에서, "본인 확인"은 마크 최초 접속 OTP에서 나눠 담당. 즉 기존 A안의 OTP 단계가 사라지는 게 아니라 **역할만 "링크 생성 → 링크 확정"으로 바뀐다.**
- **v1.1 메이저 개정 재동의 = 디스코드 통합 (사용자 결정)**. 인게임 약관 UI는 2차 웹 약관·비디스코드 경로 대비용으로 코드만 유지하고, 기본 플로우에서는 호출되지 않는다. **튕김 완화책 확정 (2026-04-19)**: 운영 공지 선행 + **재동의 유예 72h** + 유예 초과 시 **체험 월드 강등**(메인 월드 접근 차단, 재동의 완료 시 즉시 복귀). 세 레버 모두 결합 적용. **타이머 충돌 우선순위 확정 (2026-04-19)**: 48h 강퇴 유예와 72h 재동의 유예 동시 발동 시 **48h 강퇴 우선 적용** — 강퇴 타이머 소진 시 화이트리스트 제거·체험 월드 진입 모두 차단. 재동의 72h는 강퇴 미발동 유저에 한해서만 적용.

### 4. 미성년자 보호자 동의 (2026-04-19 갱신 — 만 19세 확정)
- 디스코드 모달에 "생년월일 + 보호자 동의 체크" 필드를 추가하면 기술적으로 수집은 가능하다.
- 포로 기준 = **만 19세 미만 보호자 동의 필수**(사용자 결정). Discord ToS 13세 하한과는 구간이 겹치지 않으므로 "디스코드 가입 = 13세 이상 자인 / 포로 추가 동의 = 만 19세 미만 보호자 동의 필요"로 표현만 분리하면 충돌 없음.
- 다만 보호자 동의의 법적 효력은 체크박스만으로 완결되지 않는다는 점은 A안과 동일한 한계다.

### 5. 디스코드 미가입자 진입 장벽
- "디스코드 필수"는 포로 커뮤니티 특성상 수용 가능성이 높지만, 완전 강제는 이탈 요인이다.
- 1차에는 디스코드 경로만 제공하고, 2차에서 정적 웹 페이지(세션 토큰 + 약관 동의 + 닉네임 입력) 보조 경로를 검토하는 게 안전하다. 웹 경로도 결국 OTP 확정이 필요하므로 구조는 재사용된다.

### 6. DB 스키마 조정
- `player_agreement_log.consent_method` ENUM에 `discord_first`, `ingame`, `web` 3종 유지.
- 신규 `player_whitelist(uuid NULL, mc_name, discord_user_id, agreement_version, whitelisted_at, pending_first_verify BOOL, verify_code_hash, verify_expires_at, unlinked_at NULL)`. uuid는 닉 입력 시점엔 확보, 첫 접속 시 재확인하여 불일치 시 거절.
- `player_discord_link`는 최초 접속 OTP 확정 시점에 `is_verified=true`로 INSERT, 그 전에는 INSERT 하지 않는다. UNIQUE 제약 충돌 예방.
- 화이트리스트 테이블과 링크 테이블을 분리하는 이유: "가입 신청 기록"과 "본인 확인 완료"의 생명주기가 다르기 때문.

### 7. 운영·리스크
- **봇 다운타임 = 신규 가입 차단**. 봇 프로세스 이중화 또는 최소한 systemd 자동 재시작 + 알림 훅 필요. 기존 접속자는 영향 없게 설계.
- **디스코드 BAN 유저 재가입**: 디스코드 서버 BAN은 `discord_user_id` 기준이므로 재가입해도 동일 ID. `player_whitelist.unlinked_at` 검사로 재등록 차단.
- **디스코드 서버 강퇴 = 마크 자동 제거**: 강퇴 이벤트(GuildMemberRemove) 훅에서 `unlinked_at` 기록 + Paper 측에 kick 이벤트 전파. **유예 48h 확정 (2026-04-19)** — 디스코드 재가입·재동의 시 유예 내 복귀 허용. 48h 초과 시 `player_whitelist` 엔트리 제거.
- **약관 v1.1 메이저 개정**: **디스코드 재동의로 통합 확정 (2026-04-19)**. 인게임 재동의 UI는 코드만 유지(비디스코드 경로 대비). 디스코드 재동의 미완 유저는 메인 서버 접속 차단. 기존 접속자 튕김 리스크는 운영 공지 선행 + 유예 기간(예: 72h) + 체험 모드 fall-back 중 하나로 완화.

### 8. PR 분해 & 1차 범위
기존 A안 7 PR 대비 추가/변경 포인트.
- PR-D1: 디스코드 봇 `/agree` 명령 + 약관 임베드 + 동의 로그 적재(`consent_method='discord_first'`).
- PR-D2: 모달에서 마크 닉네임 수집 + Mojang 조회 + `player_whitelist` INSERT(pending).
- PR-D3: Paper `PlayerLoginEvent`에서 `player_whitelist` 조회 + pending 시 OTP 채팅 플로우 + 확정 시 `player_discord_link` INSERT.
- PR-D4: 기존 인게임 약관 UI PR은 v1.1 재동의·웹/비디스코드 경로 대비용으로 **유지하되 호출 조건만 `discord_first` 미수집자로 한정**.
- PR-D5: GuildMemberRemove 훅 + unlink 유예 큐.

## 리스크

- 디스코드 임베드 길이 제한으로 약관 본문 전체 가시성 확보가 어렵다. 외부 링크 병행 불가피.
- "스크롤 최하단 강제 읽기" 증명력이 인게임 UI보다 약하다. 법적 증빙용으론 버튼 타임스탬프 + 본문 버전 해시 기록으로 보완.
- 봇 단일 장애점. 이중화 전까지는 다운 알림 + 수동 화이트리스트 임시 경로 필요.
- Mojang API 레이트 리밋·네임 변경 이슈.
- 디스코드 필수 강제의 수용성 문제.

## 테스트 포인트

- 동일 닉네임을 두 디스코드 계정이 각각 신청 시 후자 거절.
- pending 24h 만료 후 다른 유저가 같은 닉 신청 시 정상 처리.
- 첫 접속 시 OTP 오입력·타임아웃·재전송 케이스.
- 디스코드 강퇴 → 48h 내 복귀 시 화이트리스트 유지, 미복귀 시 제거.
- 약관 v1.1 올렸을 때 기존 접속자 인게임 재동의 UI 정상 표시, 디스코드 경로 신규 가입자는 v1.1로 자동 기록.
- 봇 다운 상태에서 기존 접속자는 영향 없음, 신규 가입만 차단되는지 확인.

## 1차 버전 범위

- PR-D1, D2, D3, D4 변경분까지만. 유예 언링크(PR-D5)·웹 보조 경로·봇 이중화는 2차.
- OTP는 6자리 숫자, TTL 10분, 재발급 3회 제한.
- 약관 본문은 디스코드 임베드 + 외부 정적 링크 2단 표시. 인게임 UI는 v1.1 대비·비디스코드 유입 대비로 코드만 유지.

**γ(하이브리드) 권장.** 약관 동의·닉네임 수집은 디스코드에서 받되, 본인 확인 OTP와 v1.1 재동의 UI는 기존 A안에서 그대로 계승한다. 근거 3줄: (1) 디스코드 단독(α)은 닉네임 도용·봇 장애 시 신규 가입 전면 차단이라는 단일 장애점이 크다. (2) A안 단독(β)은 디스코드 커뮤니티 운영 이점을 살리지 못하고 약관 가시성·재열람성도 떨어진다. (3) 하이브리드는 기존 `player_agreement_log`·`player_discord_link` 스키마를 재활용하면서 화이트리스트 테이블 1개만 추가하면 되므로 구조 충돌 없이 확장 가능하다.
