## 1. 핵심 원칙: 단일 플러그인 집중형 설계

### 의도

EmpireRPG 하나가 서버의 모든 핵심 로직과 데이터를 소유한다.

**왜 이렇게 결정했는가:**

포로 서버는 45일 시즌제 프로젝트 서버다. 여러 RPG 플러그인을 조합하는 방식(AuraSkills + MMOCore + RPGItems 등)은 플러그인 간 데이터 경계가 불명확해지고, 시즌 종료 후 다음 시즌에 데이터를 마이그레이션하기 어렵다. 반면 단일 플러그인이 모든 상태를 소유하면:

- 어떤 데이터가 어디 있는지 명확하다
- 플러그인 업데이트 충돌이 없다
- 시즌 종료 시 DB 한 군데만 백업/초기화하면 된다
- 밸런스 수치를 파일(CSV/config)로 중앙 관리할 수 있다

**다른 플러그인의 역할:**

| 플러그인 | 역할 정의 |
|---|---|
| MythicMobs | 몹/보스 스폰 트리거, 바닐라 기반 외형만 담당. 보상/패턴 로직은 EmpireRPG가 처리 |
| IridiumSkyblock | 영지 섬 생성/보호/방문 껍데기. 성장/저장고/생산 로직은 EmpireRPG가 처리 |
| Citizens | 제거 확정. NPC 기반 진행 없음 |

## 2. Bootstrap → Runtime 패턴

### 의도

각 엔진은 `Bootstrap` 클래스가 초기화를 담당하고, `Runtime` 클래스가 실제 서비스를 담는다.

```text
CombatEngineBootstrap → CombatEngineRuntime
GrowthEngineBootstrap → GrowthEngineRuntime
BossEngineBootstrap   → BossEngineRuntime
LifeEngineBootstrap   → LifeEngineRuntime
```

**왜 이렇게 결정했는가:**

플러그인 onEnable()에서 모든 초기화를 하면 순서 의존성이 꼬이고, 어디서 무엇이 초기화됐는지 추적하기 어렵다. Bootstrap/Runtime 분리는:

- 초기화 순서를 명시적으로 선언할 수 있다 (Foundation → MasterRegistry → Engines 순서)
- 각 엔진의 의존성이 Runtime 생성자에 드러난다
- 테스트 시 Bootstrap 없이 Runtime을 직접 생성할 수 있다
- 엔진 간 경계가 명확해져 기능 추가/제거가 용이하다

**초기화 순서:**

```text
1. CommonFoundationBootstrap  (DB, 로거, 설정)
2. MasterRegistryBootstrap    (CSV 마스터 데이터)
3. NpcSyncBootstrap
4. CombatEngineBootstrap
5. BossEngineBootstrap
6. GrowthEngineBootstrap
7. LifeEngineBootstrap
8. QuestAchievementBootstrap
9. OperationsQueryBootstrap
```

각 단계는 이전 단계의 Context를 주입받는다. Context 없이는 다음 단계가 시작될 수 없다.

## 3. CSV 시드 기반 마스터 데이터

### 의도

장비, 스킬, 보스, 강화 테이블 등 밸런스 수치는 `src/main/resources/seeds/*.csv` 파일에 저장하고, 서버 시작 시 DB에 upsert한다.

**왜 이렇게 결정했는가:**

밸런스 수치를 코드에 하드코딩하면 수치 변경마다 재컴파일이 필요하다. config.yml에 전부 넣으면 파일이 방대해지고 형식 유효성 검증이 어렵다. CSV 시드 방식은:

- 스프레드시트에서 작성 → CSV 저장 → 서버 재시작으로 반영 가능
- 각 행이 독립된 레코드라 검토/수정이 쉽다
- DB에 upsert되므로 운영 중 핫픽스도 CSV 교체 + reload로 처리 가능
- 외부 에디터(Excel, Google Sheets)로 밸런스 시트 관리 가능

**주요 CSV 시드 파일:**

| 파일 | 내용 |
|---|---|
| `boss_master.csv` | 보스 ID, 이름, HP, 페이즈 |
| `boss_entry_rule.csv` | 보스 입장 조건 (레벨, 그룹 크기) |
| `boss_pattern_seed.csv` | 보스 패턴 모듈 정의 |
| `state_master.csv` | 버프/디버프 상태 정의 |

## 4. 인메모리 캐시 + JSON 영속화

### 의도

플레이어 상태는 메모리 `Store`에서 읽고, 5분마다 + 퇴장 시 JSON 파일로 저장한다.

**왜 이렇게 결정했는가:**

스킬 타격, 강화 클릭 등 전투 중 발생하는 연산은 빈도가 높다. 매번 SQLite에 쓰면 I/O가 병목이 된다. 또한 45일 시즌 서버에서 PostgreSQL 같은 외부 DB를 별도로 세팅하는 것은 오버엔지니어링이다. JSON + 주기적 저장 방식은:

- 전투/강화 연산에서 DB I/O 없이 메모리만 사용
- 5분 주기 저장으로 서버 비정상 종료 시 최대 5분 손실
- JSON 파일은 사람이 직접 읽고 수정할 수 있어 운영 편의성 높음
- 시즌 종료 후 `player_data/` 디렉토리만 백업하면 데이터 보존 완료

**손실 허용 근거:**

45일 시즌 서버에서 5분 롤백은 수용 가능하다. 장기 운영 서버라면 WAL 모드 SQLite나 외부 DB 전환이 필요하다.

## 5. 전투 입력: 무기 보유 상태에서 클릭/키 조합

### 의도

핫바 스킬 슬롯 방식 없이, 무기를 들고 있을 때 4가지 입력(좌클릭/우클릭/Shift+우클릭/F)으로 스킬을 사용한다.

**왜 이렇게 결정했는가:**

핫바 슬롯 스킬 방식(1~9번 키)은 시각적으로 익숙하지만 핫바를 스킬로 채우면 아이템 공간이 부족하다. 또한 클릭 입력은 마인크래프트 기본 조작과 자연스럽게 연결된다:

- 좌클릭: 평타의 연장선인 기본기
- 우클릭: 도구 사용의 연장선인 보조기/이동기
- Shift+우클릭: 강조된 행동인 특수기
- F (오프핸드 교체): 핵심기 전용 입력

이 방식은 무기를 들고 있지 않을 때는 기본 마인크래프트 조작을 유지하므로 영지 농사나 건축 중 스킬이 오발되지 않는다.

## 6. 보스 패턴 모듈화

### 의도

보스 패턴을 개별 보스에 하드코딩하지 않고, 공통 패턴 모듈을 조합해 각 보스를 구성한다.

**왜 이렇게 결정했는가:**

1차 시즌 보스가 4개(필드보스5 + 시즌보스3 + 최종보스1)라도, 각 보스를 독립적으로 코딩하면 패턴 재사용이 없고 수정 범위가 넓다. 모듈화하면:

- 전방 강타, 돌진, 원형 폭발 등 공통 패턴은 한 번 구현 후 여러 보스에 적용
- 새 보스 추가 시 기존 모듈 조합 + 시그니처 패턴 1~2개만 추가하면 됨
- 수치(데미지, 범위, 지속시간)는 `boss_pattern_seed.csv`에서 보스별로 재정의
- 2차 시즌에서 외부 모델 교체 시 패턴 코드는 그대로 유지

**패턴과 외형 분리 원칙:**

```text
패턴 모듈 (EmpireRPG) ← 지금 구현
보스 외형 (MythicMobs + 바닐라) ← 1차
보스 외형 (ModelEngine/BetterModel/FMM) ← 2차 확장 후보
```

## 7. 영지 저장고: DB 기반 가상 저장고

### 의도

영지 저장고는 마인크래프트 인벤토리/상자가 아닌 DB에 저장되는 가상 저장고다.

**왜 이렇게 결정했는가:**

IridiumSkyblock 기반 섬에 물리적 상자를 배치하고 호퍼/케이블로 연결하는 방식은 시각적으로 익숙하지만:

- 서버 성능에 영향을 주는 tile entity가 늘어난다
- 작물 자동 수확 → 호퍼 → 상자 체인이 막히면 아이템 소실 위험
- 저장량 제한 구현이 복잡해진다 (상자 개수로 제한 불가)

DB 가상 저장고 방식은:

- 저장량을 작위 등급별로 깔끔하게 제어 가능
- 아이템 소실 위험 없음
- 웹/디스코드 봇에서 저장고 현황을 API로 바로 조회 가능
- 섬 블록 구조와 무관하게 저장고 기능 독립

## 8. HTTP API 내장

### 의도

EmpireRPG가 JDK 내장 HttpServer를 사용해 포트 8765에서 REST API를 제공한다. Discord 봇과 웹 대시보드는 이 API를 통해 데이터를 조회한다.

**왜 이렇게 결정했는가:**

외부 마이크로서비스를 별도로 운영하면 배포 복잡도가 올라간다. 45일 시즌 서버에서 Spring Boot 별도 서버를 세팅하는 것은 과도하다. JDK 내장 HttpServer는:

- 추가 라이브러리 없이 경량 API 서버를 플러그인 내에서 실행
- 게임 데이터와 API 서버가 같은 프로세스 → 네트워크 없이 메모리 참조
- Discord 봇이 인증 코드 생성, 역할 부여 큐 조회 등을 HTTP로 간단히 호출

**보안 고려:**

API 키(`api-key` 설정)로 외부 요청을 제한한다. 운영 환경에서는 API 포트를 방화벽으로 외부 차단하고 내부 네트워크에서만 접근한다.

## 9. 플레이어 플래그 시스템

### 의도

튜토리얼 완료 여부, 첫 접속 여부 등 단순 on/off 상태는 `PlayerFlagRepository`에서 key-value 형태로 DB에 저장한다.

**왜 이렇게 결정했는가:**

매번 새로운 bool 상태를 추가할 때마다 PlayerSaveData 클래스를 수정하고 마이그레이션을 작성하는 것은 번거롭다. 플래그 시스템은:

- `FlagKey` enum에 새 키를 추가하는 것만으로 새 플래그 사용 가능
- DB 스키마 변경 없이 확장 가능
- 어떤 플래그가 있는지 `FlagKey` 파일 하나에서 전부 파악 가능

## 10. 경매 시스템: 파티 내부 경매 우선

### 의도

보스 클리어 후 드랍된 고대흔적은 파티원끼리 60초 경매를 진행한 후 낙찰자가 획득한다. 솔로라면 즉시 지급한다.

**왜 이렇게 결정했는가:**

막타 보상 방식은 파티 내 막타 경쟁을 유발한다. 기여도 균등 분배는 기여를 많이 한 플레이어가 손해라고 느낀다. 파티 내부 경매는:

- 원하는 아이템에 골드를 베팅하므로 수요가 있는 플레이어가 가져감
- 파티 전투 이후 자연스러운 사회적 상호작용
- 경매 골드는 파티원에게 균등 분배 → 원하지 않는 아이템이면 골드로 이득
