# 서버 생애주기 상세 설계 (T21)

> **[STATUS: DRAFT]** — 2026-06-06. 시즌제 서버의 준비→활성→종료 운영을 구현 가능 수준으로 명세.
> 레지스트리 스키마 = [`data_model.md`](data_model.md) §2.1 `servers`. 요약 결정 = [`task.md`](task.md) §11.
> 가시성 정책 = task.md §10. 봇 관여 경계 = [`architecture.md`](architecture.md)·[`integration_contract.md`](integration_contract.md).

## 0. 경계 — 봇은 디스코드 측만 만진다
- `/서버종료`는 **게임 서버를 끄지 않는다.** 게임 시즌 인스턴스 기동/종료는 게임 호스팅 소관(봇 권위 아님, DL-133).
- 봇이 하는 것: **레지스트리 상태 전이 + 디스코드 카테고리/역할 가시성 + 명령·토글 게이팅 + 운영로그**.
- 게임 화이트리스트는 시즌 리셋 포함 **게임서버가 관리**. 봇은 디스코드 역할만 부여/회수.

## 1. 상태 모델

```
  /서버신설            /서버시작            /서버종료
없음 ───────▶ prep ───────────▶ active ───────────▶ ended (terminal)
              (준비)            (활성)              (아카이브 보존)
```

- `prep` 준비: 카테고리·역할 생성됨, 비공개. 유저 접근 불가.
- `active` 활성: 카테고리 공개(접근역할 가시), 명령·토글 동작.
- `ended` 종료: **완전 비활성**(모든 명령·토글 거부). 카테고리·역할은 **아카이브 보존**(삭제 X). 종착 상태 — 재오픈 안 함(새 시즌은 새 행).
- 모든 전이는 **운영자 명령으로만**, 전이마다 `mod_log` 적재 + `#운영로그` 게시.

## 2. 운영자 명령 (admin · `requires_permission`)

| 명령 | 입력 | 효과 | 멱등/검증 |
|---|---|---|---|
| `/서버신설` | `domain, season_no, display_name` | `servers` prep 행 생성 + 카테고리 템플릿 생성(T17) + 접근역할 생성·지정 | `UNIQUE(domain,season_no)` 중복 거부 |
| `/서버시작` | `server_id`(또는 domain+season) | prep→active. 카테고리 가시성 ON, 이모지 서버선택 패널 등록, 명령 게이팅 통과 허용 | 이미 active면 noop+안내 |
| `/서버종료` | `server_id, 사유` | active→ended. 카테고리 아카이브(숨김), 패널에서 제거, 명령·토글 차단. 카테고리·역할 보존 | 이미 ended면 noop+안내 |
| `/서버목록` | — | 전체 서버 상태(prep/active/ended) 임베드 | 조회 |
| `/서버정보` | `server_id` | 단일 서버 상세(카테고리·역할·시즌·시각) | 조회 |

- 모두 ephemeral 응답 + `#운영로그` 공개 기록(누가·언제·무엇).
- 상태 변경 명령(`신설`·`시작`·`종료`)은 **사용자 명시 요청 시 실구현**(DL-130 ⑤). 지금은 인터페이스 설계까지.

## 3. 전이별 디스코드 액션

| 전이 | 디스코드 작업 |
|---|---|
| 신설 (→prep) | 카테고리 생성(템플릿: 약관·접속정보·잡담 등) · 접근역할 생성 · 채널 권한=접근역할에게만 가시 · 접근역할은 아직 미배포(운영자/봇만) |

> 🟢 **T17 구현(2026-06-09): `modules/server_lifecycle/templates.py` `provision()`** — `/서버신설 자동생성:True`(기본)가 접근역할(`<표시명> 접근`) + 카테고리 + 채널 세트(`_DEFAULT_TEMPLATE`=공지·약관·인증·가이드·자유채팅·스크린샷·파티모집·건의-버그제보·`➕ 음성방 만들기`(임시음성 허브, T13), `_DOMAIN_TEMPLATES`로 도메인별 오버라이드 — 평소엔 공통 하나만 관리)를 **prep=비공개**(@everyone·접근역할 view=False)로 생성하고 그 ID를 `servers` 행에 기록. 채널은 카테고리 권한 상속(synced) → `/서버시작`의 카테고리 가시성 변경이 전파. 실패/중복(`UNIQUE`) 시 `cleanup()`으로 롤백. `자동생성:False` 또는 수동 `category` 지정 시 자동전개 생략. ⚠ 생성된 약관/인증 채널의 온보딩 패널 배선은 후속(ONBOARDING_SERVERS는 현행 env ID 기반).
| 시작 (prep→active) | 카테고리 가시성 ON · 서버선택 이모지 패널에 항목 추가 · `state=active` |
| 종료 (active→ended) | 카테고리를 아카이브 처리(이름 `[종료]` 프리픽스 또는 아카이브 카테고리로 이동, 가시성 제거하되 채널·역할 **보존**) · 이모지 패널에서 제거 · `state=ended, ended_at=now` |

> "아카이브" 정의: 채널 삭제 X. 가시성만 닫고 표식. 채팅 히스토리·구조 보존(§11 확정).

## 4. 게이팅 통합 — 3층 (core 헬퍼)

명령 실행 시 순서대로 통과해야 동작:
1. **접근역할(가시성)** — 디스코드 측. 운영자 Integrations 설정으로 접근역할 없으면 명령 자체가 목록에 안 보임.
2. **카테고리 가드** — `core` 헬퍼 `requires_category(domain)`: `interaction.channel.category_id`가 그 domain의 `servers.category_id`와 일치하는지. 아니면 ephemeral 거부.
3. **서버 상태** — `core` 헬퍼 `requires_server_active(domain)`: 해당 domain의 active 시즌 행이 있는지. 없으면(prep/ended) 거부.

> 둘 다 접근한 유저가 엉뚱한 카테고리에서 호출 → 2층에서 차단(§10 확정). 종료 서버 → 3층에서 차단.

## 5. 새 시즌 오픈/종료 런북 (운영)

```
오픈:
 1. /서버신설 rpg 3 "RPG 시즌3"      → prep (카테고리·역할 생성)
 2. (게임서버 측 시즌3 인스턴스 준비 — 봇 외부)
 3. /서버시작 <id>                    → active (유저 접근 가능)
종료:
 4. /서버종료 <id> "시즌3 종료"        → ended (아카이브, 명령·토글 차단)
```

- 새 **시즌** = 위 절차(코드 변경 0, 레지스트리 데이터 토글).
  - ⚠ 단서: 새 시즌이 **다른 게임서버 인스턴스(다른 API URL/키)**면 `.env` 항목 추가 + 재기동은 필요(코드 0이나 *config 0은 아님*). `servers.api_env_key`로 클라이언트가 시즌별 베이스 URL을 선택하도록 설계 — 현행 `rpg_api`의 전역 `PORO_API_URL` 단일 참조는 멀티시즌 시 확장 필요.
- 새 **게임 종류** = 위 + `modules/<도메인>/` 모듈 코드 추가 + EXTENSIONS 등록(불가피).

## 6. 미확정
- `prep→ended` 취소 경로 허용 여부(신설했다가 안 열기).
- 종료 시 온보딩 역할(`인증유저` 등) 처리 — 게임 화이트리스트는 게임서버 시즌 리셋 소관. 디스코드 역할 회수 여부는 별도 결정.
- 이모지 서버선택 패널 갱신 = 자동(봇이 메시지 재구성) vs 수동.
- 아카이브 시즌 누적 정리(`/서버정리`, N시즌 경과) = 운영 가이드.
