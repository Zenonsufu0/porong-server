# AGENTS.md

## 목적
이 저장소에서 Codex는 포로 서버의 기획, 시스템 설계, 구현 보조, 코드 수정, 구조 정리를 할 때
기존 기준을 존중하면서 작은 단위의 검토 가능한 결과를 만드는 것을 목표로 한다.

## 최우선 원칙
- 이미 확정된 포로 서버 기준을 먼저 따른다.
- 관련 없는 파일은 건드리지 않는다.
- 한 번에 큰 변경보다 작은 단위 변경을 우선한다.
- 기존 구조를 이유 없이 뒤집지 않는다.
- 확실하지 않은 내용은 추측으로 확정하지 않는다.
- 충돌하는 기준이 있으면 임의로 선택하지 말고 먼저 충돌 지점을 정리한다.
- 변경된 설정은 poro_master_planning.md에 반영한다.

## 현재 포로 서버 기준 요약
- 장비는 T1 / T2 2단계 구조다.
- T1은 성장용, T2는 최종 파밍 장비다.
- T2는 재료 제작형 구조를 우선한다.
- T2 세트는 5종이며 각 세트의 플레이스타일 정체성을 유지한다.
- 강화는 최대 25강, 파괴/하락 없음 구조다.
- 강화는 고정 수치와 옵션 수치를 모두 올린다.
- 잠재는 3줄 구조이며 등급, 이탈 잠재 규칙을 가진다.
- 공용각인은 직업각인 1개 + 공용각인 3개 구조다.
- 상태 체계는 버프 / 디버프(상태이상, 제어, 표식류) 분류를 따른다.
- 피해 증가 카테고리는 일반 / 태그 / 조건부 3개로 분리한다.
- 태그는 최대 2개까지 적용한다.
- 전사 3무기(대검, 태도, 창)의 기본 방향과 각인 정체성은 이미 정리되어 있다.
- 운영 목표는 장비/각인/스탯/통계 데이터를 DB에 저장하고 웹/디스코드 봇에서도 조회 가능하게 만드는 것이다.

## Codex 역할
Codex는 아래 작업에서 특히 유용하다.
- 기획안을 구현 단위로 쪼개기
- 기존 코드베이스 기준으로 실제 구현 리스크 찾기
- DB/스키마/엔티티/조회 구조 초안 제시
- API/서비스/저장 구조 연결점 찾기
- 작은 범위의 코드 수정
- 테스트 포인트, 회귀 위험, 누락된 예외 케이스 검토

Codex는 아래 행동을 함부로 하지 않는다.
- 확정된 설정 뒤집기
- 요청 없는 대규모 리팩터링
- 요청 없는 라이브러리 추가
- 요청 없는 배포/운영 설정 수정
- 요청 없는 전역 포맷팅
- 비밀값 파일(.env, secrets 등) 읽기 또는 출력

## 작업 시작 전 체크리스트
작업을 시작하기 전에 항상 아래를 먼저 확인한다.

1. 이번 요청이 기획 / 밸런스 / 데이터 구조 / 구현 / 디버깅 중 어디에 속하는가
2. 기존 포로 서버 기준과 충돌하는가
3. 현재 요청 범위를 벗어나는 대규모 수정이 필요한가
4. 정말 수정이 필요한 파일이 무엇인가
5. 검증 가능한 형태로 끝낼 수 있는가

## 작업 원칙
- 항상 최소 diff를 우선한다.
- 수정 전에 짧게 계획을 세운다.
- 가능하면 관련 파일만 읽고 수정한다.
- 구현보다 설계가 먼저 필요한 작업이면 먼저 구조를 정리한다.
- 코드 수정 후에는 검증 방법을 남긴다.
- 테스트 실패가 현재 요청과 무관하면 별도로 분리해서 보고한다.
- 운영자 시점에서 데이터 조회와 통제 가능성을 항상 고려한다.

## 포로 서버 도메인별 세부 원칙

### 1. 기획/기능 설계
- 기능은 개요, 목표, 핵심 규칙, 범위, 비범위, 오픈 질문으로 정리한다.
- 1차 버전과 확장 버전을 구분한다.
- 이미 있는 시스템 위에 얹는 방향을 먼저 고려한다.

### 2. 전투/밸런스
- 일반 / 태그 / 조건부 피해 증가 3분류를 함부로 무시하지 않는다.
- 상태 체계와 태그 체계를 깨지 않는다.
- 대검/태도/창의 기존 정체성을 우선 반영한다.
- 세트효과는 수치 압도보다 플레이스타일 완성도를 우선 본다.
- 수치 제안 시 폭주 가능성, 악용 조합, 일반 유저 체감도 함께 본다.

### 3. 경제/성장
- 일반 유저 기준 성장 체감이 유지되는지 본다.
- 강화, 잠재, 세트 파밍, 재료 수급 구조가 서로 충돌하지 않는지 본다.
- 특정 콘텐츠만 강제되는 병목 구조를 피한다.
- 상업화 검토 시 성장재료 직판보다 치장/편의 중심 철학을 우선한다.

### 4. 데이터 구조
- 장비, 강화, 잠재, 각인, 스탯, 전투 통계는 분리 저장을 우선 고려한다.
- 웹/디스코드 봇/운영툴에서 재사용 가능한 조회 구조를 우선한다.
- 운영자가 직접 통제 가능한 수치와 통계 뷰를 고려한다.
- 무결성과 확장성을 동시에 본다.

### 5. 구현
- 거대한 시스템을 한 번에 완성하려 하지 않는다.
- 1차 구현에서 꼭 필요한 것과 나중에 미룰 것을 구분한다.
- 디버깅 가능한 구조, 테스트 가능한 구조를 우선한다.
- 복잡한 시스템은 스키마 → 서비스 → 조회 → UI/봇 순으로 분해한다.

## 수정 우선순위
항상 아래 순서대로 판단한다.

1. 정확성
2. 기존 기준 준수
3. 최소 수정
4. 검증 가능성
5. 유지보수성
6. 확장성

## 금지 사항
- 요청 없는 대규모 파일 이동
- 요청 없는 네이밍 전면 교체
- 요청 없는 전체 구조 재설계
- 요청 없는 민감정보 접근
- 기존 기준을 무시한 “새 시스템 갈아엎기”식 제안
- 근거 없는 수치 확정
- 문서와 코드의 범위를 한 번에 크게 섞는 행동

## 애매할 때 행동
- 확실하지 않으면 먼저 가정과 불확실성을 분리해서 쓴다.
- 여러 선택지가 있으면 하나만 밀지 말고 장단점을 짧게 비교한다.
- 구현 난이도가 높으면 1차 단순안과 확장안을 나눠 제안한다.
- 기존 맥락이 강한 프로젝트이므로, 새 제안보다 기존 기준 정합성을 먼저 점검한다.
